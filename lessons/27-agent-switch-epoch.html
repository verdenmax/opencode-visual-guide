Agent 切换与 Epoch 替换Agent switch & epoch

整个第五部分，有一个名字被反复点到、却一直没拆——AgentReplacementBlocked。第 24 课讲纪元时说「换 agent 要走 replace、可能被 block」，第 25、26 课又两次预告「这是压轴的硬骨头」。现在，到了啃它的时候。这一课，也是第五部分的收官，回答一个听起来简单、做起来微妙的问题：当你在会话中途换了一个 agent，那份为旧 agent 建立的上下文纪元，该怎么办？为什么「替换」有时会被「阻挡」？答案，会把第 21 课那个看似不起眼的 unavailable ≠ removed 原则，和整套系统「宁可阻塞、绝不残缺」的品格，一齐照亮。读完它，你不仅看懂一个边角机制，更会摸到 opencode 对待「根基性变更」时那种近乎固执的审慎。

为什么「换 agent」是个需要小心处理的硬骨头？因为它要求从零重建一份完整的基线。第 24 课讲过：同一个 agent 的环境演进，走 reconcile（增量补）；可换了 agent，旧基线的「人设」整个作废，必须 replace——重新观察所有源、从头拼一份新基线。而「从头拼一份完整基线」这件事，有个苛刻的前提：所有源此刻都得观察得到。万一某个源恰好临时挂了（第 21 课那个 unavailable）呢？这一课的全部张力，就在这个「万一」里。

🏔️ 生活类比
接着第 24 课登山的比喻：换 agent，就是换一支全新的登山队，新队伍得有自己的大本营（新纪元）。可要扎一个新大本营，你得先做一次完整的环境勘测——海拔、坐标、天气、补给点，一样都不能少（重新观察所有源）。现在设想：就在你准备扎新营的当口，GPS 临时没信号了（一个源 unavailable）。你会怎么办？一个负责任的领队绝不会凭半个坐标就硬扎营——那等于让新队伍住进一个位置都没标准的营地，迟早出事。他会暂停扎营，等 GPS 恢复；在这期间，旧队的大本营依然有效、照常待命。这就是 AgentReplacementBlocked 的全部精神：与其建一个缺了一块的新基线，不如阻塞、等待、保留旧的。一次推迟的交接，远胜一个有缺口的新营。

reconcile 还是 replace：变化的两种量级

先把第 24 课那个分岔再钉清楚，因为它是这一课的起点。每次 prepare 对表时，stored.agent（存档里的 agent）和当前 agent 一比，决定走哪条路：

同 agent → reconcile

「人设」没变，只是环境飘了点。在旧基线上算增量、补一句 update 即可。常态运转。

换 agent → replace

「人设」整个换了，旧基线作废。必须重新观察所有源、从头拼一份新基线。改朝换代。

这里的关键差异，在于两条路对「源观察不到」的容忍度，截然不同。reconcile 很宽容：某个源此刻 unavailable？没关系，它保住上一份快照就行——反正基线还在，缺这一轮的观察无伤大雅（这正是第 21 课 unavailable ≠ removed 的常态体现）。但 replace 没有这份从容：它要拼的是一份全新的基线，没有「上一份快照」可以保底——每个源都必须此刻、当场、新鲜地观察到，才能拼出一份完整、自洽的新基线。一个能容忍「暂时看不到」，一个不能——区别就在这里炸开。

停下来体会这个对比，你会更懂「上下文有没有连续性」这件事的分量。reconcile 之所以能容忍一次缺席，是因为它站在一份已有基线的肩膀上——旧快照就是它的安全网，这一轮没读到，下一轮补上就是，基线始终完整。而 replace 是要另起炉灶，它脚下没有任何旧基线托底，是真正的「从零开始」。「从零开始」最怕的就是「开局缺料」：增量更新缺一笔，无非晚补一轮；但奠基时缺一块，那是把残缺焊进了地基，往后每一轮都带着这个缺口。所以同样是「一个源没读到」，在增量场景里是小事、在奠基场景里是大事——opencode 对这两种场景区别对待，正是看透了「打地基」和「添砖加瓦」根本不是一回事。

为什么要 block：拒绝残缺的新基线

于是 SystemContext.replace 的逻辑，就有了那道关键的闸门。它重新观察所有源，然后做一个判断：

// 简化自 system-context/index.ts 的 replaceObservation
if (entries.some((e) =>
    e._tag === "Unavailable" && getSnapshot(previous, e.key) !== undefined))
  return { _tag: "ReplacementBlocked" }   // ← 曾经有、现在却看不到 → 阻塞
return { _tag: "ReplacementReady", generation: ... }  // 全都观察到了 → 放行

读这个判断，妙处全在那个条件：某个源现在 Unavailable，而它在旧快照里曾经是有的。这两个条件缺一不可。如果一个源本来就没有（旧快照里也没它），那它现在没有，无所谓；但如果一个源本该有、此刻却临时观察不到，那就危险了——硬拼一份新基线，等于悄悄把这个源从新 agent 的世界里抹掉了。新 agent 会以为「这里不是 git 仓库」「没有项目指令」，而其实只是那一下没读到。opencode 的选择斩钉截铁：这种情况下，宁可 ReplacementBlocked、阻塞这次切换，也绝不交付一份缺了源的新基线。

❌ 将就：照拼不误

源没读到？当它没有，照样拼新基线。新 agent 被悄悄喂了一份残缺的世界——以为「这里不是 git 仓库」，做出错误决策。错误隐蔽、难查。

✅ 阻塞：等齐再建

有源暂缺就 block，保留旧纪元、等它恢复再重试。代价是切换稍延，收益是新基线永远完整可信。宁慢勿错。

这道闸门，是第 21 课那个 unavailable ≠ removed 原则的终极兑现。当时我们说「暂缺不等于删除」，听着像句无关痛痒的设计洁癖；现在你看到了它真正的牙齿：正因为系统从一开始就严格区分「暂时看不到」和「真的没有」，它才能在换 agent 这个最关键的当口，做出「等一等」而非「将就一下」的正确判断。一个早早埋下的原则，在六课之后，于一个最微妙的场景里，结出了它最重要的果实。这就是好设计的复利：当初那点克制，此刻成了不必再想的底气。

fence：让切换原子、一致

无论替换走向哪个结局，都还有最后一道纪律要守：这次切换必须是原子的、一致的——绝不能切到一半被另一个并发操作搅乱。守这道纪律的，是贯穿始终的 revision 版本号，外加一个叫 fence（栅栏）的核对动作。

fence 在写入前会再核对一遍：当前存档里的 agent，还是不是我以为的那个？revision 版本号，还是不是我读到时的那个？只要有一样对不上，就说明在我盘算这次切换的工夫里，已经有别人动过手了——于是它 die 出一个 RevisionMismatch/AgentMismatch，外层的 retryRevisionMismatch 接住、退让重试。这套「比对版本、不符就重来」的乐观并发，确保了即便有两处几乎同时想切 agent，最终也只会有一次干净地落定，绝不会留下一个半切、错乱的纪元。（顺带一提：fence 实际是在「被阻挡 / 没变化」这类不写入的分支上守门；而真正写新基线的那条路，由 replace 自己事务里的 WHERE revision = 预期值 把关——殊途同归，都是拿 revision 当令牌。）

把 block 和 fence 合起来看，你会发现 opencode 对「换 agent」这件事，守着两道防线：一道防残缺（block：源没齐就不建新基线），一道防错乱（fence：有人插手就退让重来）。两道防线指向同一个信念——纪元这种「会话的根基性状态」，要么完整、一致地建立，要么干脆不动；绝不容许一个半吊子、自相矛盾的中间态存在。这正是贯穿整个第四、五部分的那条主线（先记账、原子提交、串行排空、版本守护……）在「换 agent」这个收官场景里的最后一次回响。

退一万步说，这一课真正教给你的，或许不是 AgentReplacementBlocked 这个具体机制，而是它背后那种对「中间态」的零容忍。纵观第四、五部分，你会发现 opencode 反复在做同一件事：但凡涉及「根基性的状态变更」——admit 一个 prompt、推进一个纪元、切换一个 agent——它都坚持「要么完整成功、要么干净不变」，绝不留下一个「成功了一半」的尴尬残局。这种执拗，正是「可靠系统」和「能跑的脚本」之间，那道最深的分野：脚本只关心「正常情况能不能跑通」，而可靠系统，把大量心力花在「异常情况下，能不能优雅地什么都不破坏」。AgentReplacementBlocked，不过是这份执拗，在第五部分谢幕时，留下的最后一个注脚。

一句话总结第五部分：opencode 把「模型每轮该看到怎样一份世界」，做成了一套可组合、可增量、可持久、可演进的「上下文纪元」系统——它和会话引擎共享同一条事件时间线，宁可阻塞也不残缺，宁可重试也不错乱。引擎（第四部分）已经在跑，决定「引擎该读什么世界」的大脑（第五部分）也已讲透。接下来的第六部分，我们将潜入更底层——LLM 协议层：那个把模型的请求与响应，翻译成各家供应商（Anthropic、OpenAI、Gemini…）各自方言的精密适配系统。

🔬 源码细节

prepareOnce 里「换 agent 被 block」那一支，把这一课的逻辑钉得清清楚楚：

// 简化自 session/context-epoch.ts 的 prepareOnce
const result = stored.replacement_seq === null && !replacingAgent
  ? yield* SystemContext.reconcile(value, snapshot)   // 同 agent：增量
  : yield* SystemContext.replace(value, snapshot)     // 换 agent：重建
if (result._tag === "ReplacementBlocked" && replacingAgent) {
  yield* fence(db, sessionID, agent, stored.revision)        // 守一致
  return yield* new AgentReplacementBlocked({          // 抛类型化错误
    sessionID, previous: stored.agent, current: agent })
}

留意那个 AgentReplacementBlocked 错误带的字段：previous（原来的 agent）和 current（想换成的 agent）。它不是一句含糊的「切换失败」，而是把「谁想换成谁、却没换成」说得明明白白——又一次第 5、8、20 课「错误是值、信息齐全」那套世界观的体现。上层接到这个错误，能清楚知道「是从 A 换 B 被挡了」，从而决定是稍后重试、还是提示用户。连「被阻挡」这种边角情况，都被收进类型、带齐信息——这是整套系统对「失败也要讲清楚」的一贯执着。

🎯 本课要点
  • 换 agent 走 replace（而非 reconcile）：旧基线的「人设」作废，必须重新观察所有源、从头拼一份新基线（第 24 课）。
  • 关键差异：reconcile 容忍源 unavailable（保住旧快照）；replace 不能容忍——新基线没有「旧快照保底」，每个源都得当场新鲜观察到。
  • 为什么 block：若某源「曾经有、现在却 unavailable」，硬拼新基线等于悄悄抹掉它。opencode 宁可 ReplacementBlocked、保留旧纪元、等重试，也绝不交付残缺基线——第 21 课 unavailable ≠ removed 的终极兑现。
  • fence + revision：写入前核对 agent/版本，不符则退让重试——保证切换原子、一致，绝无半切错乱的纪元。
  • 两道防线（防残缺的 block + 防错乱的 fence）指向同一信念：纪元要么完整一致地建立，要么不动——不容许半吊子的中间态。

All through Part 5, one name was repeatedly invoked yet never unpacked — AgentReplacementBlocked. Lesson 24 on epochs said "switching agent takes replace, may be blocked"; Lessons 25 and 26 twice teased "this is the finale's hard bone." Now, time to crack it. This lesson, also Part 5's finale, answers a question that sounds simple but is subtle to do: when you switch an agent mid-session, what becomes of the context epoch built for the old agent? Why is "replacement" sometimes "blocked"? The answer illuminates, at once, Lesson 21's seemingly-unremarkable unavailable ≠ removed principle and the whole system's character of "rather block than be incomplete." Reading it, you don't just understand an edge mechanism but touch opencode's near-stubborn prudence toward "foundational changes."

Why is "switching agent" a hard bone needing careful handling? Because it demands rebuilding a complete baseline from scratch. Lesson 24 covered: a same agent's environment evolution takes reconcile (increment); but switch the agent and the old baseline's "persona" is wholly void, requiring replace — re-observe all sources, stitch a new baseline from scratch. And "stitch a complete baseline from scratch" has a strict precondition: all sources must be observable right now. What if a source happens to be temporarily down (Lesson 21's unavailable)? This lesson's whole tension lives in that "what if."

🏔️ Analogy
Continuing Lesson 24's mountaineering metaphor: switching agents is switching to a wholly new climbing team, and the new team needs its own base camp (new epoch). But to pitch a new base camp, you must first do a complete survey of the surroundings — altitude, coordinates, weather, supply points, not one missing (re-observe all sources). Now imagine: just as you're about to pitch the new camp, the GPS temporarily loses signal (a source unavailable). What do you do? A responsible leader would never pitch camp on half a coordinate — that's housing the new team at a site with no standard location, trouble sooner or later. They'd pause pitching, wait for GPS to recover; meanwhile, the old team's base camp stays valid, still on standby. That's the whole spirit of AgentReplacementBlocked: rather than build a new baseline missing a piece, block, wait, keep the old. A delayed handover beats a new camp built on incomplete data.

reconcile or replace: two magnitudes of change

First nail down Lesson 24's fork again, as it's this lesson's starting point. Each prepare reconcile, stored.agent (the archive's agent) compared with the current agent decides which road:

same agent → reconcile

The "persona" unchanged, just the environment drifted a bit. Compute the increment on the old baseline, append an update. Normal operation.

switch agent → replace

The "persona" wholly changed, the old baseline void. Must re-observe all sources, stitch a new baseline from scratch. Changing of the guard.

The key difference is the two roads' tolerance for "a source unobservable" is utterly different. reconcile is forgiving: a source unavailable right now? Fine, it keeps the last snapshot — the baseline still stands, missing this round's observation does no harm (exactly Lesson 21's unavailable ≠ removed in normal operation). But replace has no such ease: it stitches a brand-new baseline, with no "last snapshot" to fall back on — every source must be observed now, on the spot, freshly to stitch a complete, self-consistent new baseline. One can tolerate "temporarily unseen," one can't — the difference explodes right here.

Pause on this contrast and you'll better grasp the weight of "whether context has continuity." reconcile can tolerate one absence because it stands on the shoulders of an existing baseline — the old snapshot is its safety net, miss this round and catch up next, the baseline always complete. But replace starts a new fire, with no old baseline underfoot, a true "from scratch." "From scratch" most fears "missing material at the open": an increment missing one entry just catches up a round later; but laying a foundation missing a piece welds the gap into the bedrock, carrying that hole every round after. So the same "a source unread" is trivial in the increment scenario, grave in the foundation scenario — opencode treating the two differently is exactly seeing through that "laying a foundation" and "adding bricks" are fundamentally not the same thing.

Why block: refusing an incomplete new baseline

So SystemContext.replace's logic has that crucial gate. It re-observes all sources, then makes a judgment:

// simplified from replaceObservation in system-context/index.ts
if (entries.some((e) =>
    e._tag === "Unavailable" && getSnapshot(previous, e.key) !== undefined))
  return { _tag: "ReplacementBlocked" }   // ← once present, now unseen → block
return { _tag: "ReplacementReady", generation: ... }  // all observed → proceed

Reading this judgment, the cleverness is all in the condition: a source is now Unavailable, AND it was present in the old snapshot. Both conditions are indispensable. If a source never existed (not in the old snapshot either), its absence now is fine; but if a source should be there yet is temporarily unobservable, that's dangerous — forcing a new baseline would silently erase this source from the new agent's world. The new agent would think "this isn't a git repo," "there are no project instructions," when really that one fetch just failed. opencode's choice is resolute: in this case, rather ReplacementBlocked, blocking this switch, than deliver a new baseline missing a source.

❌ make do: stitch anyway

Source unread? Treat it as absent, stitch the new baseline anyway. The new agent is silently fed an incomplete world — thinks "not a git repo," makes wrong decisions. The error is hidden, hard to trace.

✅ block: build only when complete

A source missing → block, keep the old epoch, retry when it recovers. The cost is a slightly delayed switch; the gain is a new baseline always complete and trustworthy. Slow over wrong.

fence: making the switch atomic, consistent

Whichever outcome the replacement heads toward, there's one last discipline to keep: this switch must be atomic and consistent — never half-switched then disrupted by another concurrent operation. Keeping it is the revision version number running throughout, plus a checking move called fence.

fence, before writing, double-checks: the agent in the current archive, is it still the one I thought? The revision version, still the one I read? If even one mismatches, it means while I was planning this switch, someone else already acted — so it dies with a RevisionMismatch/AgentMismatch, caught by the outer retryRevisionMismatch, which backs off and retries. This optimistic concurrency of "compare versions, redo on mismatch" ensures that even if two places try to switch agents almost simultaneously, only one cleanly settles in the end, never leaving a half-switched, garbled epoch. (By the way: fence actually guards the no-write branches like "blocked / unchanged"; the path that really writes the new baseline is guarded by replace's own transactional WHERE revision = expected check — different routes, same idea: revision as the token.)

Put block and fence together and you find opencode keeps two defenses for "switching agents": one against incompleteness (block: don't build a new baseline if sources aren't all there), one against garbling (fence: someone interfered, back off and redo). Both defenses point to one belief — an epoch, this "foundational state of the session," is either established completely and consistently, or not touched at all; never is a half-baked, self-contradictory middle state permitted. This is exactly that main line running through all of Parts 4 and 5 (persist first, atomic commit, serial drain, version guarding…) echoing one last time in the finale scenario of "switching agents."

Stepping way back, what this lesson truly teaches you may not be the concrete mechanism AgentReplacementBlocked but the zero tolerance for "middle states" behind it. Across Parts 4 and 5, you'll find opencode repeatedly doing the same thing: wherever a "foundational state change" is involved — admitting a prompt, advancing an epoch, switching an agent — it insists "either complete success, or clean no-change," never leaving an awkward "half-succeeded" wreck. This stubbornness is exactly the deepest divide between a "reliable system" and a "script that runs": a script cares only "can it run in the normal case," while a reliable system spends great effort on "in the abnormal case, can it gracefully break nothing." AgentReplacementBlocked is but this stubbornness's last footnote, left as Part 5 takes its bow.

const result = stored.replacement_seq === null && !replacingAgent
  ? yield* SystemContext.reconcile(value, snapshot)   // same agent: increment
  : yield* SystemContext.replace(value, snapshot)     // switch agent: rebuild
if (result._tag === "ReplacementBlocked" && replacingAgent) {
  yield* fence(db, sessionID, agent, stored.revision)        // guard consistency

}

Note the fields the AgentReplacementBlocked error carries: previous (the original agent) and current (the agent wanted). It's not a vague "switch failed" but states plainly "who wanted to switch to whom, but didn't" — again Lessons 5, 8, 20's worldview of "errors are values, fully informed." The upper layer, receiving this error, knows clearly "the switch from A to B was blocked," and can decide to retry later or prompt the user. Even a corner case like "being blocked" is folded into the type, fully informed — this is the whole system's consistent insistence that "failure too must be stated clearly."

🎯 Key points
  • Switching agent takes replace (not reconcile): the old baseline's "persona" is void, requiring re-observing all sources and stitching a new baseline from scratch (Lesson 24).
  • The key difference: reconcile tolerates a source unavailable (keeps the old snapshot); replace can't — a new baseline has no "old snapshot fallback," every source must be freshly observed on the spot.
  • Why block: if a source is "once present, now unavailable," forcing a new baseline silently erases it. opencode rather ReplacementBlocked, keeping the old epoch, awaiting retry, than deliver an incomplete baseline — the ultimate cash-out of Lesson 21's unavailable ≠ removed.
  • fence + revision: double-check agent/version before writing, mismatch → back off and retry — ensuring the switch is atomic, consistent, never a half-switched garbled epoch.
  • Two defenses (block against incompleteness + fence against garbling) point to one belief: an epoch is either established completely and consistently, or not touched — no half-baked middle state permitted.
