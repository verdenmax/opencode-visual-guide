流式事件与缓存Streaming & caching

这一课是 M6 的「打结」课——它把好几条横跨全书的伏线，在这里系成结实的扣。第一个结：第 28 课讲「反腐层」时，我们看了 core 送出去的规范语 LLMRequest；这一课补上回来的另一半——规范事件流 LLMEvent，也就是六种协议的 stream.step 千辛万苦要翻译成的那套统一词汇，也是第 17 课 agent 循环真正消费的东西。第二个结更妙：第 24 课 Context Epoch 拼命维持「基线前缀稳定」，第 30 课 Anthropic 有「4 个缓存断点预算」——这一课的 cache-policy.ts 会告诉你，那 3 个自动缓存断点具体打在哪、为什么打在那，从而把「稳定前缀」和「省钱」之间最后一根线，当着你的面接通。

所以这一课有两个主题，恰好对应一来一回：回来的「规范事件流」（LLMEvent，反腐层的入站半边）和出去时的「自动缓存策略」（cache-policy，把稳定前缀缓存住）。两者看似不相干，其实共享一个深层主题——把分散在各课的设计意图，收束到一处具体的代码里兑现。读完你会有种「原来前面那些苦心都是为了此刻」的畅快，也会更佩服这套设计在「分层埋线、最后收束」上的耐心。

🎬 生活类比
想象模型的回复是一场现场直播，而 LLMEvent 就是这场直播的标准化导播信号。无论后台用的是哪家的摄像机（协议），导播台打出来的信号格式永远统一：「正文开始」→「正文片段、片段、片段」→「正文结束」→「要调一个工具了」→「工具参数一段段来」→「工具调用完整了」→……→「全片完，附带用时统计」。楼上的总导演（agent 循环）只看这套标准信号，从不需要知道后台是索尼还是佳能的机器。而缓存这件事，就像直播前给场地布好的固定机位：开场那一大段背景介绍（系统提示、工具说明）每场都一样，与其每次重新架设，不如把这段固定机位缓存下来——一场直播里主持人来回走动（工具往返）很多次，可只要开头那段布景不变，就能一直复用，省下大把重新架设的成本。

回来的规范事件流：LLMEvent 的统一词汇

第 28 课我们说，反腐层有一进一出：出去的是 LLMRequest，回来的是 LLMEvent 流。前者第 28 课看过了，后者一直没正面展开——它就定义在 schema/events.ts 里。打开一看，LLMEvent 是一个有 16 个成员的标签联合，但它们其实归成几族，逻辑非常清晰：

step-start / step-finish
一个推理步骤的边界（呼应第 20 课的有界步数）
text-start / -delta / -end
正文文字：开始、一段段增量、结束
reasoning-start / -delta / -end
推理过程：同样的 start/delta/end 三段式
tool-input-start / -delta / -end
工具参数：跨帧到齐的那串 JSON（第 29 课的主角）
tool-call / tool-result / tool-error
完整工具调用 / 结果 / 出错
finish / provider-error
整轮结束（带 Usage 用量）/ 供应商错误

这套词汇的设计透着一股流式的味道：凡是会「一点点来」的东西（正文、推理、工具参数），都拆成 start → delta…delta → end 的三段式。这正是第 29 课那台状态机吐出的形态——还记得吗？工具参数通过一串 tool-input-delta 跨帧累积，攒齐了才有 tool-call。把一轮典型回复的事件流摊开，你会看到一条清晰的时间线：

step-start本步开始
text-start → delta×N → text-end模型先说一段话
tool-input-start → delta×N → tool-input-end参数 JSON 一段段到齐
tool-call一个完整工具调用成形
step-finish本步结束（可能进入下一步）
finish整轮结束，附 Usage（输入/输出/缓存 token）

关键认知：这 16 种事件，就是六种协议的「最大公约数」。Anthropic 的 content_block_delta、OpenAI 的 choices[].delta、Gemini 的流式 part、Bedrock 的二进制帧——千差万别，但 stream.step 最终都把它们翻译成这同一套 LLMEvent。于是楼上的 agent 循环（第 17 课）只需认识这 16 种事件，就能驱动任何一家模型。第 28 课画的那道「翻译墙」，出站一侧是 LLMRequest、入站一侧就是 LLMEvent——到这一课，墙的两面才算都看全了。墙内是清一色的规范事件，墙外是六种方言的喧哗，而那道墙，就是 stream.step 这台翻译机。也正因为楼上只认这 16 种事件，opencode 想接入「第七家、第八家」模型供应商时，agent 循环一行都不用改——新供应商只需把自己的流式输出翻译成这套既定词汇即可。统一的事件词汇，既是对内的契约，也是对外扩展的接口。

自动缓存：把那段「不变的开头」缓存住

第二个主题，是 cache-policy.ts——一个只有一百来行、却把好几课串起来的小文件。先回顾动机：第 30 课讲过 Anthropic 的提示缓存（在某些块上打 cache_control，缓存住从开头到该标记的前缀），还讲过「最多 4 个断点」的预算。但断点到底该打在哪几个位置？——这就是 cache-policy 要回答的。它的默认策略 "auto" 打三个断点：

最后一个工具定义
markLastTool：工具列表通常整轮不变
最后一段系统提示
markLastSystem：系统提示也通常整轮不变
最新的用户消息
latest-user-message：一轮的「分水岭」

为什么偏偏是这三处？源码注释一语道破，而且这正是本课最该打通的那个结。想想第 17 课的 agent 循环：用户发一条消息后，一个回合会「炸开」成许多次 assistant↔tool 的往返——模型调工具、看结果、再调工具……每一次往返，都要把整段前缀（工具定义 + 系统提示 + 直到这条用户消息的全部历史）重新发给模型。这段前缀在整个回合里岿然不动。于是把缓存断点打在「最新用户消息」这个边界上，就意味着：这一回合内每一次工具往返的 API 调用，都命中同一段缓存前缀。一次缓存写入，被回合内 N 次往返反复命中——省的钱是成倍的。

用户消息缓存断点打在这（前缀 = 工具+系统+历史，到此为止）
往返1模型调工具A → 重发整段前缀（命中缓存！）
往返2看到结果、调工具B → 又重发前缀（再次命中）
往返N…一个回合内反复命中同一段缓存前缀…

到这里，第 24 课那根线终于接通了：Context Epoch 拼死维持「基线前缀稳定」，图的就是让这段缓存前缀持续命中。前缀一旦被中途改写，缓存即刻失效，回合内后续每次往返都得按全价重算。所以「稳住前缀」不是洁癖，是真金白银的省钱手段——cache-policy 在这一层把断点打对位置，Context Epoch 在那一层把前缀守住不变，两者一上一下，合谋成就了 prompt 缓存的高命中率。一个在第 24 课埋下的「为什么要稳住前缀」，到第 34 课才图穷匕见。

🔬 源码细节

为什么缓存默认就开（undefined → "auto"）？注释给了一笔精算账：

// 简化自 cache-policy.ts 的注释与默认
const AUTO = { tools: true, system: true, messages: "latest-user-message" }

// 账：Anthropic 5 分钟缓存，写入 = 1.25x 基础价、读取 = 0.1x。
// 只要 5 分钟内复用一次，就已经回本（1.25 + 0.1 < 2）。
// 而一个工具回合里要复用很多次 → 稳赚。所以默认就开。

缓存写入比不缓存还贵一点点（1.25x），但读取便宜到几乎免费（0.1x）。只要前缀在 5 分钟内被复用哪怕一次，这笔买卖就划算；而一个工具回合动辄十几次往返，复用次数远不止一次——所以 opencode 把缓存设成默认开启（源码 cache-policy.ts 里只给了这笔成本账，没提任何外部框架）。还有个工程细节值得一提：markMessageAt 故意不用 .map() 而用 slice + 下标赋值，注释说得很实在——长对话每次请求都要跑它，.map() 的闭包派发和整数组复制「在性能剖析里会冒头」。连这种微小的热路径开销都被盯着，是生产级代码的底色，也是这套缓存逻辑敢于「默认开启」的底气所在。

各家缓存机制不同：一个克制的「跳过」

最后一处巧思，藏在一行 RESPECTS_INLINE_HINTS 里。cache-policy 这套「在 part 上打内联标记」的玩法，只对 Anthropic 和 Bedrock 有效——因为只有它们的线缆格式认内联的 cache_control 标记。OpenAI 和 Gemini 的缓存机制根本不同：

供应商缓存机制cache-policy 怎么做
Anthropic / Bedrock显式内联标记 cache_control打断点（本课主角）
OpenAI隐式前缀缓存（自动，无需标记）整段跳过
Gemini隐式 + 带外 CachedContent API整段跳过

所以 applyCachePolicy 开头第一句就是：不在 RESPECTS_INLINE_HINTS 里的，直接原样返回、整套策略跳过。注释解释得很坦诚：给 OpenAI/Gemini 打内联标记「无害但无意义」——它们压根不看这些标记，自有一套隐式缓存。这是个很克制的设计：不强行用一套机制套所有人，而是承认各家缓存哲学的根本差异，该出手时（Anthropic/Bedrock）出手、该闭嘴时（OpenAI/Gemini）闭嘴。这恰好再次印证整个 M6 的主旋律——差异交给适配层吸收，而不是硬塞进一个统一抽象里假装它们都一样。真正好的抽象，懂得在哪里该统一、也懂得在哪里该放手。

注意这条流水线和第 30 课的衔接：cache-policy 在协议编码之前跑一遍，把该缓存的 part 标上 CacheHint；然后协议的 lowering 层（第 30 课那个带 Breakpoints{remaining,dropped} 计数器的 cacheControl）再把这些 hint 落成真正的 cache_control 标记、并守住 4 个名额的上限。一个决定「打哪」、一个执行「打上且不超额」，职责分得干干净净——cache-policy 是战略（缓存哪几处最划算），protocol 的断点预算是战术（在硬约束内落实）。两课至此严丝合缝地咬合上。

🗺️ 宏观图景

这一课把 M6（乃至前几个 part）的几根线收成了扣：

  • LLMEvent = 反腐层的入站半边（schema/events.ts，16 成员）：step / text / reasoning / tool-input 的 start-delta-end 三段式 + tool-call/result/error + finish(带 Usage) + provider-error。六种协议的 stream.step 全翻译成它，第 17 课 agent 循环只消费它。补全了第 28 课「翻译墙」的入站一面。
  • 自动缓存的 3 个断点（cache-policy.ts 的 "auto"）：最后一个工具定义 + 最后一段系统提示 + 最新用户消息——把「整回合不变的稳定前缀」缓存住。
  • 接通第 24 课：断点打在「最新用户消息」边界，使回合内每次 assistant↔tool 往返都命中同一段缓存前缀；这正是 Context Epoch 死守「基线前缀稳定」的真正目的——稳定 = 可缓存 = 省钱。
  • 缓存默认开：写 1.25x、读 0.1x，5 分钟内复用一次即回本，工具回合复用多次稳赚。各家机制不同（Anthropic/Bedrock 显式内联；OpenAI/Gemini 隐式，cache-policy 整段跳过）。

M6 至此只差最后一课。第 35 课讲模型解析与 GitHub Copilot provider：opencode 怎么从 models.dev 目录认出「某个模型该走哪条 Route、配齐哪几件」，以及 Copilot 这个「既是供应商又自带认证体系」的特殊存在。讲完它，整个「core 说规范语 → 适配器翻方言 → 传输层送达 → 事件流回来 → 缓存省钱」的 LLM 闭环就彻底合龙了。

🎯 本课要点
  • LLMEvent（schema/events.ts，16 成员）是反腐层的入站规范词汇：step-start/finish、text/reasoning/tool-input 各自的 start-delta-end 三段式、tool-call/result/error、finish（带 Usage）、provider-error。六种协议 stream.step 都翻译成它，agent 循环（L17）只消费它——补全第 28 课「翻译墙」的入站半边。
  • 三段式反映流式：会「一点点来」的内容（正文/推理/工具参数）都拆成 start→delta…→end，正是第 29 课状态机的吐出形态。
  • cache-policy.ts 的 "auto" 打 3 个断点：最后工具定义 + 最后系统提示 + 最新用户消息（markLastTool/markLastSystem/markMessages），缓存「整回合不变的稳定前缀」。
  • 接通 Context Epoch（L24）：断点打在「最新用户消息」边界，回合内每次 assistant↔tool 往返都命中同一前缀；故 Context Epoch 死守「基线前缀稳定」的真正目的就是保住缓存命中。缓存默认开（写 1.25x/读 0.1x，5 分钟内一次复用即回本）。
  • RESPECTS_INLINE_HINTS = {anthropic-messages, bedrock-converse}：只有这俩认内联标记；OpenAI（隐式前缀缓存）、Gemini（隐式 + 带外 CachedContent）整段跳过策略。体现「差异交给适配层吸收，而非假装统一」。

First knot: when lesson 28 covered the "anti-corruption layer," we saw the canonical language LLMRequest core sends out; this lesson fills in the other half coming back—the canonical event stream LLMEvent, the unified vocabulary that the six protocols' stream.step labor to translate into, and exactly what lesson 17's agent loop actually consumes. The second knot is even neater: lesson 24's Context Epoch works hard to keep the "baseline prefix stable," lesson 30's Anthropic has a "4 cache-breakpoint budget"—this lesson's cache-policy.ts tells you exactly where those 3 auto cache breakpoints land and why, thereby connecting, right before your eyes, the last wire between "stable prefix" and "saving money."

🎬 Analogy
Imagine the model's reply is a live broadcast, and LLMEvent is this broadcast's standardized director's signal. Whatever cameras (protocols) the backstage uses, the signal the director's desk outputs is always uniform: "body starts" → "body fragment, fragment, fragment" → "body ends" → "about to call a tool" → "tool args arrive in segments" → "tool call complete" → … → "show over, with timing stats." The chief director upstairs (agent loop) watches only this standard signal, never needing to know if the backstage runs Sony or Canon. And caching is like fixed camera positions set up before the broadcast: the long opening backdrop (system prompt, tool descriptions) is the same every show, so rather than re-rigging each time, cache that fixed-position shot—within one broadcast the host walks back and forth (tool round-trips) many times, but as long as the opening backdrop is unchanged, it can be reused throughout, saving the cost of re-rigging.

The canonical event stream back: LLMEvent's unified vocabulary

Lesson 28 said the anti-corruption layer has an out and a back: out goes LLMRequest, back comes the LLMEvent stream. We saw the former in lesson 28; the latter was never directly unfolded—it's defined in schema/events.ts. Open it and LLMEvent is a tagged union with 16 members, but they group into a few families, very cleanly:

step-start / step-finish
a reasoning step's boundary (echoing lesson 20's bounded steps)
text-start / -delta / -end
body text: start, segment-by-segment delta, end
reasoning-start / -delta / -end
reasoning: same start/delta/end triple
tool-input-start / -delta / -end
tool args: the JSON arriving across frames (lesson 29's star)
tool-call / tool-result / tool-error
complete tool call / result / error
finish / provider-error

This vocabulary's design has a distinctly streaming flavor: anything that "arrives bit by bit" (body, reasoning, tool args) is split into a start → delta…delta → end triple. This is exactly the shape lesson 29's state machine emits—remember? tool args accumulate across frames via a run of tool-input-delta, and only once assembled is there a tool-call. Lay out a typical turn's event stream and you'll see a clear timeline:

step-startthis step begins

tool-input-start → delta×N → tool-input-endarg JSON assembles segment by segment
tool-calla complete tool call takes shape
step-finishthis step ends (may enter the next)
finishwhole turn ends, with Usage (input/output/cache tokens)

Key realization: these 16 events are the "greatest common divisor" of the six protocols. Anthropic's content_block_delta, OpenAI's choices[].delta, Gemini's streaming parts, Bedrock's binary frames—wildly different, but stream.step ultimately translates them all into this same LLMEvent set. So the agent loop upstairs (lesson 17) need only know these 16 events to drive any model. The "translation wall" lesson 28 drew has LLMRequest on the outbound side and LLMEvent on the inbound side—only by this lesson are both faces of the wall fully seen. Inside the wall is uniform canonical events, outside is the clamor of six dialects, and the wall itself is the translation machine stream.step. Precisely because upstairs knows only these 16 events, when opencode wants to add a "seventh, eighth" model provider, the agent loop doesn't change a line—a new provider need only translate its streaming output into this established vocabulary. A unified event vocabulary is both an internal contract and the interface for outward extension.

Auto caching: cache that "unchanging opening"

The second theme is cache-policy.ts—a file of only a hundred-some lines that strings several lessons together. First recall the motive: lesson 30 covered Anthropic's prompt caching (mark some blocks with cache_control, caching the prefix from the start to that marker), and the "at most 4 breakpoints" budget. But where exactly should the breakpoints land?—that's what cache-policy answers. Its default policy "auto" places three breakpoints:

last tool definition
markLastTool: the tool list usually stays fixed all turn
last system part
markLastSystem: the system prompt also usually fixed all turn
latest user message
latest-user-message: a turn's "watershed"

Why exactly these three? The source comment nails it, and this is the very knot this lesson should tie. Think of lesson 17's agent loop: after a user sends one message, a single turn "explodes" into many assistant↔tool round-trips—the model calls a tool, sees the result, calls another tool… each round-trip must resend the entire prefix (tool definitions + system prompt + all history up to this user message) to the model. That prefix stays rock-steady through the whole turn. So placing the cache breakpoint at the "latest user message" boundary means: every tool round-trip's API call within this turn hits the same cached prefix. One cache write, repeatedly hit by the turn's N round-trips—the savings are multiplied.

user msgcache breakpoint placed here (prefix = tools+system+history, up to here)
round-trip 1model calls tool A → resend the whole prefix (cache hit!)
round-trip 2sees result, calls tool B → resend prefix again (hit again)
round-trip N…repeatedly hitting the same cached prefix within one turn…

Here, lesson 24's thread finally connects: Context Epoch works to death to keep the "baseline prefix stable" precisely to keep this cached prefix hitting continuously. Once the prefix is rewritten midway, the cache instantly invalidates, and every subsequent round-trip in the turn must recompute at full price. So "hold the prefix steady" isn't fastidiousness, it's a hard-cash money-saving measure—cache-policy places breakpoints right at this layer, Context Epoch guards the prefix unchanged at that layer, the two conspiring, one above one below, to achieve prompt caching's high hit rate. A "why hold the prefix steady" planted in lesson 24 finally shows its hand in lesson 34.

🔬 Source Detail

Why is caching on by default (undefined → "auto")? The comment gives a precise reckoning:

// simplified from cache-policy.ts comments and defaults
const AUTO = { tools: true, system: true, messages: "latest-user-message" }

// Math: Anthropic 5-min cache, write = 1.25x base, read = 0.1x.
// A single reuse within 5 minutes already wins (1.25 + 0.1 < 2).
// And a tool turn reuses many times → sure win. So on by default.

A cache write costs slightly more than no cache (1.25x), but a read is nearly free (0.1x). As long as the prefix is reused even once within 5 minutes, the deal pays off; and a tool turn easily has a dozen-plus round-trips, reusing far more than once—so opencode sets caching on by default (the source cache-policy.ts gives only this cost math, citing no external framework). One more engineering detail worth noting: markMessageAt deliberately avoids .map() and uses slice + index assignment, the comment frank about it—long conversations run it on every request, and .map()'s closure dispatch and whole-array copies "show up in profiling." Even such tiny hot-path overhead is watched—the hallmark of production code, and the confidence behind this cache logic daring to be "on by default."

Different vendors, different cache mechanisms: a restrained "skip"

One last touch of cleverness hides in a line, RESPECTS_INLINE_HINTS. cache-policy's whole "mark inline hints on parts" approach works only for Anthropic and Bedrock—because only their wire formats recognize the inline cache_control marker. OpenAI's and Gemini's cache mechanisms are fundamentally different:

VendorCache mechanismWhat cache-policy does
Anthropic / Bedrockexplicit inline marker cache_controlplace breakpoints (this lesson's star)
OpenAIimplicit prefix caching (automatic, no marker)skip the whole pass
Geminiimplicit + out-of-band CachedContent APIskip the whole pass

So applyCachePolicy's very first line is: anything not in RESPECTS_INLINE_HINTS returns as-is, the whole policy skipped. The comment explains candidly: marking inline hints for OpenAI/Gemini is "harmless but pointless"—they don't look at these markers at all, having their own implicit caching. It's a very restrained design: not forcing one mechanism on everyone, but acknowledging each vendor's fundamentally different caching philosophy, acting when it should (Anthropic/Bedrock) and shutting up when it should (OpenAI/Gemini). This again confirms all of M6's main melody—let the adapter layer absorb differences, rather than cramming them into one unified abstraction pretending they're all the same. A truly good abstraction knows where to unify and where to let go.

Drawn as a pipeline, applyCachePolicy's "fill-in-the-blanks" nature is clear—it fills only the cells the caller left empty, never overwriting existing manual markers:
