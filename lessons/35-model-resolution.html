模型解析与 Copilot providerModel resolution & Copilot

这是 M6 的收官之课。前面六课，我们从「反腐层」的总图（L28）一路走到协议骨架（L29）、四家方言（L30~32）、传输六旋钮（L33）、事件流与缓存（L34）。但有个最朴素的问题一直悬着没答：用户在界面上敲下一个模型名（比如 anthropic/claude-sonnet），opencode 是怎么从这个字符串，一路解析到「该走哪条 Route、配哪组旋钮、用什么价钱、上限多少」的？这一课就补上这条「从一个名字到一条配好的链路」的解析线。而压轴登场的，是 GitHub Copilot——一个把前六课所有伏笔都收齐的「终极样本」：它复用 OpenAI 的协议、却自带一套独特的认证，是「六旋钮架构」威力的最后一次、也是最漂亮的一次展示。

这一课会让你看清两件事。其一，opencode 不把模型的事实写死在代码里，而是去拉取 models.dev 这个社区维护的「模型大全」——某模型支持不支持工具调用、上下文多大、输入输出多少钱、缓存读写各打几折……全来自这个外部目录。其二，catalog 把这份目录变成可查的内存结构，让「模型名 → 一条配好的 Route」这次解析得以发生。把这两件事看懂，整个「core 说规范语 → 适配器翻方言 → 传输送达 → 事件回来 → 缓存省钱」的 LLM 闭环，就彻底合龙了。

📖 生活类比
想象 opencode 是一家「万国翻译公司」，前六课我们招齐了翻译（协议）、配好了物流（传输）。可客户上门只会说一句：「我要找『克劳德·十四行诗』这位专家。」公司怎么从这个名字，立刻调出此人的完整履历卡：他会哪些活（能力）、收费多少（成本）、一次最多接多长的稿子（上下文上限）、走哪条物流线（Route）？靠的是一本随时更新的行业黄页（models.dev）——公司自己不硬记每个专家的资料，而是定期从这本公开黄页同步，新专家一上线就自动能查到。而 Copilot 这位「专家」尤其特别：他干的活和 OpenAI 那位一模一样（同一套协议），但进他家门得用一张特殊的门禁——先拿你的 GitHub 工牌去换一张当天有效的临时通行证，再凭通行证进门。同样的翻译本事，套一层独特的门禁系统，这就是 Copilot 的与众不同。

models.dev：一本不写死在代码里的「模型大全」

第一个关键设计：opencode 不在源码里硬编码任何模型的事实。「claude-sonnet 支持 20 万 token 上下文」「gpt 输入每百万 token 多少钱」这类信息，是会随厂商发版不断变化的——若写死在代码里，每次模型更新都得改代码发版，苦不堪言。opencode 的解法是去拉取 models.dev：一个社区维护的、覆盖几乎所有 LLM 的公开模型目录。models-dev.ts 里的 Model schema，就定义了每个模型被追踪的事实：

能力
attachment / reasoning / temperature / tool_call：支不支持附件、推理、调温、工具
cost
输入/输出/缓存读/缓存写 的单价（第 34 课算账用的就是它！）
limit
context 上下文上限、input/output token 上限
modalities
输入/输出支持的模态：text/audio/image/video/pdf
family / release_date / status
模型家族、发布日期、alpha/beta/deprecated 状态

看到那个 cost 里的「缓存读 / 缓存写」单价了吗？这又是一根接通的线——第 34 课讲缓存默认开时，那句「写 1.25x、读 0.1x、复用一次即回本」的精算，算盘珠子就拨在这些来自 models.dev 的单价上。模型大全不只是一本「通讯录」，它还是缓存决策、用量计费的事实地基。models-dev.ts 通过 HTTP 把这份目录拉下来、缓存住，并在刷新时广播一个 models-dev.refreshed 事件：

models.dev
社区公开目录
HTTP 拉取 →
本地缓存
解析成 Model/Provider
刷新时 →
广播 refreshed
models-dev.refreshed 事件

把易变的事实外置到一个会自我更新的目录，这样新模型一上线，opencode 往往一行代码都不用改就能认得它——这和第 31 课「OpenAI 兼容白嫖生态」是同一种「把变化挡在代码之外」的智慧，只不过那次挡的是「协议的变化」，这次挡的是「模型事实的变化」。再往深想一层，这其实是一种责任的转移：模型的价格、上限、能力，本该由「最了解它们、且最频繁更新它们」的人来维护——而那不是 opencode 的开发者，是 models.dev 这个社区。opencode 只负责「消费」这份目录，把「维护事实」的脏活累活，交给了一个专门做这件事的上游。不重复造轮子，也不重复维护事实，这是更高一层的复用。

catalog：从「模型名」到「一条配好的 Route」

拉下来的 models.dev 目录是原始数据，要真正能用，还得变成一个可快速查询的内存结构——这就是 catalog.ts（CatalogV2）干的事。它把目录组织成一张两层的表：

模型名
anthropic/claude-sonnet
拆 →
providerID + modelID
catalog 查 →
ModelInfo
能力/成本/上限/走哪条 Route
→
配好的 Route
第 33 课那六个旋钮

catalog 的核心是一个 providers: Map<ProviderID, ProviderRecord>，每个 ProviderRecord 里又有一个 models: Map<ModelID, ModelInfo>。于是「一个模型名」的解析，就是两次字典查找：先按 providerID 找到供应商、再按 modelID 找到具体模型信息。查不到？catalog 给了两个类型化的错误，把失败的两种情形分得清清楚楚：

ProviderNotFoundError

连供应商都没找到（比如模型名拼错了 provider 段）。第一层查找就失败。

ModelNotFoundError

供应商有，但它名下没有这个模型。第二层查找失败，错误里带上 providerID + modelID。

注意这种设计的体贴：它不是返回一个含糊的 null 让上层去猜「到底哪一步出了错」，而是明确告诉你「是供应商没找到，还是供应商有、但这个模型没找到」。两种失败的修复方式截然不同——前者你得检查 provider 名、后者你得检查 model 名——把它们用不同的错误类型分开，上层就能给出精准的提示。这种把失败分得清清楚楚的设计，呼应了整本书反复出现的「让错误自己说清自己是谁」。这一步解析的产物 ModelInfo，正握着「这个模型该走哪条 Route」的钥匙——而 Route，就是第 33 课那个六旋钮的组合。至此，一个用户输入的模型名，终于接上了一条从协议到传输、配置完整的链路。

Copilot：复用 OpenAI 协议，自带一套门禁

压轴的，是 GitHub Copilot 这个特殊供应商。它特殊在哪？翻开 github-copilot/copilot-provider.ts，第一眼你就会看到它 import 了 OpenAICompatibleChatLanguageModel 和 OpenAIResponsesLanguageModel——它的协议，整个复用了 OpenAI 那套（第 31 课的 Chat 与 Responses）！这意味着「请求怎么编码、响应怎么解码」，Copilot 一个字都不用重写。那它到底「特殊」在哪？特殊在认证：

协议复用 OpenAI Chat / Responses（第 31 课），零重写
端点指向 Copilot 自己的 baseURL
认证GitHub OAuth 工牌 → 换取短期 Copilot token → Bearer 头
头部附加 Copilot 特有的自定义 headers

看出来了吗？Copilot 正是第 33 课「六个正交旋钮」的最后一次、也是最漂亮的一次实战演示：protocol 旋钮拨到「OpenAI 兼容」（复用）、endpoint 旋钮拨到 Copilot 的地址、auth 旋钮拨到「GitHub 换 token」这套独特流程。把它和 OpenAI 官方并排，哪几个旋钮变了、哪几个没动，一目了然：

旋钮OpenAI 官方GitHub Copilot
protocolopenai-chat / responses同上（整套复用！）
endpointapi.openai.comCopilot 自己的 baseURL
auth固定 API keyGitHub 工牌换短期 token
transport / framingHTTP / SSE同上

它和第 31 课的「openai-compatible 只改端点」是同一招的升级版——这次不光改端点，还换了一套认证，但协议依然原封不动地复用。Copilot 的认证比一般的 API key 复杂：你不能直接拿一个固定密钥，而要先用 GitHub 的 OAuth 工牌，去换取一个有时效的 Copilot 专用 token，再用它做 Bearer 认证。正因为 auth 早被设计成一个独立旋钮，这套「先换证、再进门」的特殊流程，才能整个塞进 auth 旋钮里、丝毫不惊动协议层。假如当初认证逻辑和协议编解码缠在一起，Copilot 这种「同协议、异认证」的家伙就会逼你把 OpenAI 协议又抄一遍——而正交分解让它优雅地化解于无形。用一个最棘手的真实供应商，给整个 M6 的设计哲学盖了章。

为什么要绕这么一圈「先换证、再进门」？因为这套令牌交换背后是一种更安全的设计：你的 GitHub OAuth 凭证是长期的、贵重的，不该被反复发到推理端点上；而换来的 Copilot token 是短期、限定用途的，即便泄露，危害也被时效和权限死死框住。这正是「最小权限 + 短时效凭证」的安全惯例。opencode 要做的，无非是把这套交换流程封装进 auth 旋钮——上层只管说「我要用 Copilot」，底下「拿工牌换通行证」的来回，被 auth 这一层悄悄消化掉，协议层和 agent 循环全程无感。安全性的复杂，被恰当地隔离在了它该待的那一层，没有渗进别处。

🔬 源码细节

Copilot 复用 OpenAI 模型、只换认证头的痕迹，在 copilot-provider.ts 里清晰可见：

// 简化自 github-copilot/copilot-provider.ts
import { OpenAICompatibleChatLanguageModel } from "./chat/..."
import { OpenAIResponsesLanguageModel } from "./responses/..."

// 协议复用 OpenAI；只在 headers 上做文章
const headers = {
  ...(apiKey && { Authorization: `Bearer ${'$'}{apiKey}` }),  // 换来的 Copilot token
  ...options.headers,                                  // Copilot 特有头
}
provider.chat = createChatModel        // ← 仍是 OpenAI Chat 那套
provider.responses = createResponsesModel  // ← 仍是 OpenAI Responses 那套

整个文件做的事，本质上就是「拿 OpenAI 的两个语言模型，套一层自己的 headers 和 baseURL」。chat 和 responses 两个方法直接指向 OpenAI 兼容的实现——协议层完全没动。所有「Copilot 味」都浓缩在 headers 和 baseURL 这两处配置里，而非代码逻辑里。这是一个极好的「差异应该体现为配置、而非分支」的范例：Copilot 与 OpenAI 的不同，没有变成散落在协议代码里的一堆 if (isCopilot)，而是收敛成「换一组 headers、换一个 baseURL」这样干净的配置差异。把差异关进配置的笼子，协议代码才能永远只有一份。

🗺️ 宏观图景 · M6 全景收官

这一课补完模型解析，整个 M6（LLM 协议层）的闭环正式合龙。回望这一整个 part：

  • L28 反腐层：core 只说规范语（LLMRequest 出 / LLMEvent 回），适配器砌一道翻译墙。
  • L29 协议骨架：每个协议都填同一张「两栏表」（body 请求侧 + stream 响应侧状态机）。
  • L30~32 四家方言：Anthropic（块+缓存预算）、OpenAI（Chat/Responses 双协议 + 24 行白嫖生态）、Gemini+Bedrock（改名方言 + 二进制传输/套娃状态机）。
  • L33 传输六旋钮：Route = id/protocol/endpoint/auth/transport/framing，正交可组合。
  • L34 事件与缓存：LLMEvent（入站规范词汇，补全翻译墙）+ cache-policy（把稳定前缀缓存住，接通 Context Epoch）。
  • L35 解析与 Copilot：models.dev（外置的模型大全，事实地基）→ catalog（模型名→配好的 Route）；Copilot 复用 OpenAI 协议、只换 endpoint+auth，给六旋钮架构盖章。

一句话串起来：用户敲一个模型名，catalog 凭 models.dev 的事实把它解析成一条配好六旋钮的 Route；core 用规范语发请求，Route 的协议把它翻成方言、传输送上线、framing 切帧、协议再把响应解码回规范 LLMEvent 流交给 agent 循环，全程的缓存则靠 cache-policy 与 Context Epoch 联手压低成本。七课走下来，你会发现整个 LLM 层没有一处是「为了炫技」，每一层抽象都在解一个真实的痛点——这正是值得反复回味的工程之美。下一个 part（M7）将转向工具系统——agent 循环手里那些真正「干活」的工具（读写文件、跑命令、搜索……）是怎么定义、执行、隔离的。LLM 是 agent 的「嘴和脑」，工具则是它的「手」。

🎯 本课要点
  • models.dev = 外置的社区模型目录（models-dev.ts 经 HTTP 拉取、缓存、发 models-dev.refreshed）：每个 Model 记录能力（attachment/reasoning/tool_call…）、cost（输入/输出/缓存读写单价，第 34 课算账的数据源）、limit（上下文/输入/输出上限）、modalities。把易变的模型事实挡在代码之外，新模型常常零改动即可识别。
  • catalog（catalog.ts CatalogV2）= 解析器：providers: Map<ProviderID, {provider, models: Map<ModelID, ModelInfo>}>。模型名经两次字典查找 → ModelInfo（含走哪条 Route）；查不到给类型化的 ProviderNotFoundError / ModelNotFoundError。
  • 解析全链路：模型名 → 拆 providerID+modelID → catalog 查到 ModelInfo → 第 33 课的六旋钮 Route → 可发起 stream。
  • Copilot = 六旋钮架构的终极演示（github-copilot/copilot-provider.ts）：协议整套复用 OpenAI（Chat+Responses，零重写），只拨动 endpoint（Copilot baseURL）和 auth（GitHub OAuth 工牌 → 换短期 Copilot token → Bearer + 自定义头）两个旋钮。
  • 差异是配置、不是分支：Copilot 与 OpenAI 的不同收敛成「换 headers + baseURL」的配置差异，而非协议代码里的 if (isCopilot)。正因 auth 早被设计成独立旋钮，「先换证再进门」的特殊认证才能不惊动协议层。M6 闭环至此合龙，下一站 M7 工具系统。

1. 为什么 opencode 不把「某模型支持多大上下文、输入输出多少钱」写死在代码里，而去拉取 models.dev？
  1. 因为 models.dev 跑得更快
  2. 这些事实随厂商发版不断变化；外置到一个社区维护、会自更新的目录，新模型常常零改动即可识别，把「易变事实」挡在代码之外
  3. 纯粹为了多一个网络依赖
  4. 因为代码里不能写数字
看答案与解析 点击展开
答案：B。模型的上限、价格、能力随厂商发版频繁变动，写死就得每次改代码发版。opencode 改去拉取 models.dev——一个社区维护、覆盖几乎所有 LLM 的公开目录，缓存住并在刷新时发 models-dev.refreshed 事件。这是「把变化挡在代码外」的智慧（和第 31 课 OpenAI 兼容白嫖生态同源），更是把「维护事实」的脏活交给最了解它们的上游社区——不重复维护事实，是更高层的复用。
2. catalog 解析「模型名」失败时，给的是 ProviderNotFoundError 和 ModelNotFoundError 两个不同错误，而非一个 null。好处是？
  1. 为了让代码更长
  2. 两个错误纯粹是冗余
  3. 错误越多显得越专业
  4. 明确区分「供应商没找到」和「供应商有但模型没找到」两种失败——它们修复方式不同，分开后上层能给精准提示
看答案与解析 点击展开
答案：D。catalog 是两层字典：先按 providerID 查供应商、再按 modelID 查模型。两种失败的修复截然不同——ProviderNotFound 你得检查 provider 段、ModelNotFound 你得检查 model 段（错误还带上 providerID+modelID）。用不同错误类型把它们分开，而非返回含糊的 null 让上层猜「哪步错了」，上层就能给出精准提示。这呼应全书「让错误自己说清自己是谁」。
3. GitHub Copilot 作为「特殊供应商」，它的「特殊」主要体现在六个旋钮的哪一个上？
  1. auth（和 endpoint）：协议整套复用 OpenAI Chat/Responses，只换端点 + 一套「GitHub 工牌换短期 token」的独特认证
  2. framing：它用二进制分帧
  3. 六个旋钮全都和别人不同
  4. protocol：它发明了一套全新协议

答案：A。copilot-provider.ts 直接 import OpenAICompatibleChat/Responses 语言模型——协议零重写。它的「特殊」全在 auth：不能用固定 API key，要先拿 GitHub OAuth 工牌换一个有时效的 Copilot token 再做 Bearer 认证（外加自定义头），endpoint 也指向 Copilot 自己。这是第 33 课六旋钮的最佳实战：protocol 旋钮复用、只拨 endpoint+auth。差异收敛成「换 headers+baseURL」的配置，而非协议里的 if(isCopilot)。

This is M6's finale. Over the prior six lessons, we walked from the "anti-corruption layer" overview (L28) through the protocol skeleton (L29), four dialects (L30–32), transport's six knobs (L33), event stream and caching (L34). But one plain question lingered unanswered: when a user types a model name in the UI (say anthropic/claude-sonnet), how does opencode resolve that string all the way to "which Route to ride, which knobs to set, what price, what limits"? This lesson fills in that "from a name to a configured pipeline" resolution thread. And taking the final bow is GitHub Copilot—an "ultimate sample" that collects all six prior lessons' threads: it reuses OpenAI's protocol yet brings its own distinctive auth, the last and most beautiful display of the "six-knob architecture"'s power.

This lesson clarifies two things. One, opencode doesn't hardcode model facts in code, but fetches models.dev, a community-maintained "model encyclopedia"—whether a model supports tool calling, how big its context is, the price of input/output, the discount on cache read/write… all from this external catalog. Two, catalog turns this directory into a queryable in-memory structure, enabling the "model name → a configured Route" resolution. Understand these two and the whole LLM loop—"core speaks canonical → adapters translate dialects → transport delivers → events come back → caching saves money"—is fully closed.

Imagine opencode is a "universal translation company." Over six lessons we hired the interpreters (protocols) and arranged logistics (transport). But a client walks in saying just: "I want the expert 'Claude Sonnet.'" How does the company instantly pull up this person's complete résumé card from that name: what skills they have (capabilities), their fees (cost), the longest manuscript they take at once (context limit), which logistics line they use (Route)? Via a constantly-updated industry yellow pages (models.dev)—the company doesn't memorize each expert's data, but periodically syncs from this public directory, so a new expert is auto-queryable the moment they come online. And Copilot, this "expert," is especially special: the work he does is identical to OpenAI's (same protocol), but entering his door requires a special access pass—first take your GitHub badge to exchange for a temporary same-day pass, then enter with that pass. The same translation skill, wrapped in a distinctive access system—that's what sets Copilot apart.

models.dev: a "model encyclopedia" not hardcoded in code

The first key design: opencode hardcodes no model's facts in source. Information like "claude-sonnet supports 200K-token context," "gpt costs how much per million input tokens"—these keep changing with vendor releases; hardcoded in code, every model update would mean changing code and shipping a release, agonizing. opencode's solution is to fetch models.dev: a community-maintained public model catalog covering nearly all LLMs. The Model schema in models-dev.ts defines each model's tracked facts:

capabilities
attachment / reasoning / temperature / tool_call: supports attachments, reasoning, temperature, tools?
cost
unit prices for input/output/cache-read/cache-write (the very data lesson 34's reckoning used!)
limit
context cap, input/output token caps
modalities
input/output supported modalities: text/audio/image/video/pdf
family / release_date / status
model family, release date, alpha/beta/deprecated status

See those "cache read / cache write" prices in cost? Another connected thread—lesson 34's "write 1.25x, read 0.1x, one reuse pays off" reckoning was computed against exactly these prices from models.dev. The model encyclopedia isn't just an "address book," it's the factual bedrock for caching decisions and usage billing. models-dev.ts fetches this catalog over HTTP, caches it, and broadcasts a models-dev.refreshed event on refresh:

Externalizing volatile facts into a self-updating directory means a new model coming online is often recognized by opencode without a line of code changed—the same "keep change out of the code" wisdom as lesson 31's "OpenAI-compatible free-rides the ecosystem," only that time it blocked "protocol changes," this time "model-fact changes." Deeper still, this is a transfer of responsibility: a model's prices, limits, capabilities should be maintained by "those who know them best and update them most often"—and that's not opencode's developers, it's the models.dev community. opencode only "consumes" this catalog, handing the grunt work of "maintaining facts" to an upstream dedicated to it. Don't reinvent the wheel, and don't re-maintain facts either—a higher-order reuse.

catalog: from "model name" to "a configured Route"

The fetched models.dev catalog is raw data; to be truly usable it must become a quickly-queryable in-memory structure—that's what catalog.ts (CatalogV2) does. It organizes the catalog into a two-level table:

lesson 33's six knobs

catalog's core is a providers: Map<ProviderID, ProviderRecord>, each ProviderRecord holding a models: Map<ModelID, ModelInfo>. So resolving "a model name" is two dictionary lookups: first find the provider by providerID, then the specific model info by modelID. Not found? catalog gives two typed errors that cleanly separate the two failure cases:

ProviderNotFoundError

even the provider isn't found (e.g. the provider segment of the model name is misspelled). The first lookup fails.

ModelNotFoundError

the provider exists but has no such model. The second lookup fails, the error carrying providerID + modelID.

Note this design's thoughtfulness: it doesn't return a vague null leaving the upper layer to guess "which step failed," but tells you clearly "is it the provider not found, or the provider exists but this model not found." The two failures' fixes differ entirely—the former you check the provider name, the latter the model name—and separating them with different error types lets the upper layer give precise hints. This design of cleanly separating failures echoes the book's recurring "let errors say who they are." This resolution step's output ModelInfo holds the key to "which Route this model should ride"—and a Route is lesson 33's six-knob composition. Now a user-entered model name finally connects to a complete, configured pipeline from protocol to transport.

Copilot: reuse the OpenAI protocol, bring your own gate

Taking the final bow is GitHub Copilot, a special provider. Special how? Open github-copilot/copilot-provider.ts and at first glance you'll see it imports OpenAICompatibleChatLanguageModel and OpenAIResponsesLanguageModel—its protocol reuses OpenAI's entirely (lesson 31's Chat and Responses)! Meaning "how requests encode, how responses decode," Copilot rewrites not a word. So where is it "special"? In auth:

protocolreuse OpenAI Chat / Responses (lesson 31), zero rewrite
endpointpoints to Copilot's own baseURL
authGitHub OAuth badge → exchange for a short-lived Copilot token → Bearer header
headersattach Copilot-specific custom headers

See it? Copilot is the last and most beautiful field demonstration of lesson 33's "six orthogonal knobs": the protocol knob turned to "OpenAI-compatible" (reused), the endpoint knob to Copilot's address, the auth knob to the distinctive "GitHub exchanges a token" flow. Lay it next to OpenAI official and which knobs changed, which didn't, is plain:

KnobOpenAI officialGitHub Copilot
protocolopenai-chat / responsessame (entirely reused!)
endpointapi.openai.comCopilot's own baseURL
authfixed API keyGitHub badge exchanges short-lived token
transport / framingHTTP / SSEsame

It's an upgraded version of lesson 31's "openai-compatible only changes the endpoint"—this time not only the endpoint but a whole new auth, yet the protocol is still reused untouched. Copilot's auth is more complex than a plain API key: you can't just hold a fixed secret, but must first use GitHub's OAuth badge to exchange for a time-limited Copilot-specific token, then use it for Bearer auth. Precisely because auth was designed early as an independent knob, this special "exchange first, then enter" flow can be stuffed entirely into the auth knob, disturbing the protocol layer not at all. Had auth logic been entangled with protocol codec, a "same-protocol, different-auth" creature like Copilot would force you to copy the OpenAI protocol again—and orthogonal decomposition dissolves it elegantly into nothing. Using one of the trickiest real providers, it stamps a seal on all of M6's design philosophy.

Why the whole roundabout "exchange first, then enter"? Because this token exchange reflects a more secure design: your GitHub OAuth credential is long-lived and precious, and shouldn't be repeatedly sent to inference endpoints; while the exchanged Copilot token is short-lived and purpose-limited, so even if leaked, the damage is tightly bounded by expiry and scope. This is the "least privilege + short-lived credential" security convention. All opencode does is encapsulate this exchange flow into the auth knob—the upper layer just says "I want Copilot," and the underlying "badge-for-pass" round trip is quietly digested by the auth layer, the protocol layer and agent loop oblivious throughout. The complexity of security is properly isolated in the layer where it belongs, without seeping elsewhere.

🔬 Source Detail

The trace of Copilot reusing OpenAI models and only swapping auth headers is clear in copilot-provider.ts:

// simplified from github-copilot/copilot-provider.ts
import { OpenAICompatibleChatLanguageModel } from "./chat/..."
import { OpenAIResponsesLanguageModel } from "./responses/..."

// protocol reuses OpenAI; only the headers are touched
const headers = {
  ...(apiKey && { Authorization: `Bearer ${'$'}{apiKey}` }),  // the exchanged Copilot token
  ...options.headers,                                  // Copilot-specific headers
}
provider.chat = createChatModel        // ← still the OpenAI Chat set
provider.responses = createResponsesModel  // ← still the OpenAI Responses set

What the whole file does is essentially "take OpenAI's two language models, wrap them in its own headers and baseURL." The chat and responses methods point directly to the OpenAI-compatible implementations—the protocol layer untouched. All the "Copilot flavor" is concentrated in the two pieces of configuration headers and baseURL, not in code logic. It's an excellent example of "differences should manifest as configuration, not branches": Copilot's difference from OpenAI didn't become a pile of if (isCopilot) scattered through protocol code, but converged into clean config differences like "swap a set of headers, swap a baseURL." Cage the difference in configuration, and the protocol code can forever stay single-copy.

🗺️ The Big Picture · M6 Panoramic Finale

This lesson completes model resolution, and the whole M6 (LLM protocol layer) loop formally closes. Looking back over this entire part:

  • L28 anti-corruption layer: core speaks only canonical (LLMRequest out / LLMEvent back), adapters wall off with translation.
  • L29 protocol skeleton: every protocol fills the same "two-column form" (body request side + stream response-side state machine).
  • L30–32 four dialects: Anthropic (blocks + cache budget), OpenAI (Chat/Responses dual protocol + 24-line ecosystem free-ride), Gemini+Bedrock (renamed dialect + binary transport / nesting-doll state machine).
  • L33 transport six knobs: Route = id/protocol/endpoint/auth/transport/framing, orthogonal and composable.
  • L34 events and caching: LLMEvent (inbound canonical vocabulary, completing the translation wall) + cache-policy (cache the stable prefix, connecting Context Epoch).
  • L35 resolution and Copilot: models.dev (externalized model encyclopedia, factual bedrock) → catalog (model name → configured Route); Copilot reuses the OpenAI protocol, swapping only endpoint+auth, sealing the six-knob architecture.

In one sentence: the user types a model name, catalog resolves it via models.dev's facts into a six-knob-configured Route; core sends a request in canonical, the Route's protocol translates it into dialect, transport puts it on the wire, framing cuts frames, the protocol decodes the response back into a canonical LLMEvent stream for the agent loop, and caching throughout is kept cheap by cache-policy and Context Epoch joining forces. Seven lessons in, you'll find not one part of the LLM layer is "for showing off"—every layer of abstraction solves a real pain point, which is exactly the engineering beauty worth savoring repeatedly. The next part (M7) turns to the tool system—how the tools the agent loop wields to actually "do work" (read/write files, run commands, search…) are defined, executed, isolated. The LLM is the agent's "mouth and brain," tools its "hands."

🎯 Key Takeaways
  • models.dev = an externalized community model catalog (models-dev.ts fetches via HTTP, caches, emits models-dev.refreshed): each Model records capabilities (attachment/reasoning/tool_call…), cost (input/output/cache-read/write prices, lesson 34's reckoning data source), limit (context/input/output caps), modalities. Blocking volatile model facts out of the code, a new model is often recognized with zero changes.
  • catalog (catalog.ts CatalogV2) = the resolver: providers: Map<ProviderID, {provider, models: Map<ModelID, ModelInfo>}>. A model name via two dictionary lookups → ModelInfo (incl. which Route); not found gives typed ProviderNotFoundError / ModelNotFoundError.
  • The full resolution chain: model name → split providerID+modelID → catalog finds ModelInfo → lesson 33's six-knob Route → ready to stream.
  • Copilot = the ultimate six-knob demo (github-copilot/copilot-provider.ts): protocol entirely reuses OpenAI (Chat+Responses, zero rewrite), turning only the endpoint (Copilot baseURL) and auth (GitHub OAuth badge → exchange short-lived Copilot token → Bearer + custom headers) knobs.
  • Difference is configuration, not branches: Copilot's difference from OpenAI converges into "swap headers + baseURL" config, not if (isCopilot) in protocol code. Because auth was designed early as an independent knob, the special "exchange first, then enter" auth disturbs the protocol layer not at all. M6's loop now closes; next stop M7, the tool system.
