权限系统Permissions

前五课，我们给 agent 配齐了一双强大的手：能改文件、跑命令、搜网、抓页、问人。但能力越大，越需要一根缰绳——一个能 rm -rf、能把代码推上线、能往外发请求的 agent，要是没人管，那是灾难而非助手。这一课的权限系统，就是那根缰绳。它回答一个贯穿所有工具的问题：这一步，到底准不准做？你或许已经注意到，前面每个工具的 execute 里都埋着一句 permission.assert(...)——它们全都通向这里。工具讲的是「能做什么」，权限讲的是「准做什么」；前者是能力，后者是边界，二者合起来，才是一个既有用、又可控、又敢托付的 agent。

这套系统的内核出奇地简洁——它本质是一个 allow / deny / ask 三态规则引擎：每条规则说「对某个动作、某个资源，放行 / 拒绝 / 问一下」；评估时后匹配的规则胜出，而没有规则命中时，默认是「问」。这个「默认问」的设计是整套安全模型的灵魂：agent 想做任何没被预先批准的敏感事，要么撞上一条 allow 规则、要么就得停下来等用户点头。更妙的是用户的回答有三档——once（就这次）/ always（以后都行，并记下来）/ reject（不行）——其中 always 会把决定持久化成一条规则，于是系统会随着信任的积累，问得越来越少。读懂这一课，你就明白 opencode 是怎么在「让 agent 放手干」和「让人始终在环」之间，找到那个动态平衡点的。

🛂 生活类比
把权限系统想象成一栋大楼的门禁前台 + 一本规则册。每当工人（agent）要进某个房间做点敏感事（对某资源跑某工具），都得在前台刷一下。前台翻规则册：写着「允许」就放行；写着「禁止」就拦下（而且这种房间根本不会出现在工人的楼层图上）；规则册里没提到的，前台就按铃叫经理（用户）来定夺。经理可以说三种话：「就这一次」（放行这回）、「以后都行」（放行，且前台把这条写进规则册，下次同样的事就不再打扰经理）、「不行」（拦下，还能附一句「为什么不行」捎给工人）。日子久了，规则册渐渐填满经理的常规决定，前台按铃越来越少——工人越来越顺手，而经理始终对「册子上没有的新情况」保留最终拍板权。放手，但不失控，松紧自如，这就是门禁前台的智慧。

三态规则引擎：allow / deny / ask

权限系统的内核，是一个朴素得惊人的数据结构。打开 permission/schema.ts，一条规则就三个字段：

// permission/schema.ts —— 一条规则
Rule = { action: string, resource: string, effect: "allow" | "deny" | "ask" }
Ruleset = Rule[]

action（做什么，如 edit/bash）、resource（对谁，如某文件路径）、effect（怎么办）。三个 effect 就是这套系统的全部「态度」：

allow
放行，静默执行，不打扰用户
deny
拒绝，抛 DeniedError；且这种工具在第 37 课就被「whollyDisabled」挡在菜单外
ask
问用户——发一个 permission.v2.asked 事件，停下来等回答

评估靠一个 evaluate(action, resource, ...rulesets) 函数，逻辑只有两句话，却字字关键：把各 ruleset 拼起来，findLast 找最后一条「动作和资源都通配匹配」的规则，用它的 effect；若一条都没匹配上，默认 effect: "ask"。这两句话定义了整套安全模型的两条铁律：

后匹配者胜（findLast）

规则按顺序排，越靠后越优先。可以先写一条宽泛规则、再用一条具体规则去覆盖它——和防火墙规则表一个套路。

默认即「问」

没有任何规则提到的动作，不默认放行、也不默认拒绝，而是问人。这是「安全默认」：未知 = 交给人判断。

「默认问」这一条，看似平常，实则是整套设计的定盘星。它意味着：agent 无法靠「钻规则的空子」悄悄做成一件没人批准的事——凡是规则册没明说放行的敏感操作，都会触发一次对用户的询问。新工具、新场景、边界情况……一切「未被预先考虑」的，统统落到「问一下」这个安全网里。把「未知」默认归到「问人」，而不是「放行」，是这套权限系统最重要的一个选择。反过来想就明白它的分量：如果默认是「放行」，那每出现一个设计者没预料到的新动作，agent 就可能悄无声息地做了一件你本想拦下的事；而默认「问」，把这种「漏网」的风险从根上堵死——代价不过是初期多问几句。在「安全」与「省事」之间，这套系统毫不犹豫地选了前者，再用 always 把「省事」一点点挣回来。

评估流：assert 的一次完整裁决

这套规则怎么作用到一次具体的工具调用上？看 permission.assert——前面每个工具 execute 里那句。它发起一次完整的裁决：

工具调 assert
{action, resource}
evaluate →
查 effective 规则集
findLast，未命中→ask
三态分流 →
allow→放行
deny→DeniedError
ask→发问、等回答

三条分支：deny → 直接抛 DeniedError，工具根本跑不起来；allow → 静默返回，工具照跑；ask → 发一个 permission.v2.asked 事件、挂起等待（用一个 Deferred 把这次执行暂停在原地），直到用户那边给出回答。注意整个 assert 被裹在 uninterruptibleMask 里——「问—等—答」这一串不能被中途打断，否则就可能出现「问了一半、状态错乱」的尴尬。这又是第 8 课「把关键时序原子化」的体现。

这里值得停下来，看清 deny 和 ask 其实在两个不同的关口把关——这是一种纵深防御：

deny · 菜单层（第 37 课）

一个被彻底 deny 的工具，在 materialize 时就被 whollyDisabled 滤掉，根本不出现在发给模型的清单里——模型连「有这个工具」都不知道。

ask · 调用层（本课）

工具在菜单上、模型能点，但每次真要执行时 assert 先发问——执行前的最后一道闸。

这两级的分工很讲究：「绝不允许」的，在源头就让模型看不见（不给它「想都不要想」的机会，也省下解释和拒绝的来回）；「视情况而定」的，留在菜单上、临门一脚再问。最干净的拒绝是让对方根本不知道有这个选项（呼应第 37 课），而最灵活的把关是临场征求同意——deny 与 ask 各司其职，一个管「能不能有」、一个管「这次行不行」。

用户的回答（reply）分三档，每一档都对应一种不同的「未来」：

once就这一次放行；下次同样的事，还会再问
always放行 + 把这条写进持久化规则（saved.add → SQLite，按 project 作用域）；以后同样的事直接 allow，不再问
reject拒绝这次 → RejectedError；若附了反馈则 CorrectedError（把「为什么不行」捎回给模型）

这里 reject 的两种形态值得多看一眼。光说「不行」（RejectedError）是纯拦截；但若用户在拒绝时附一句理由，就变成 CorrectedError——而这句理由会顺着错误回传给模型。这一笔很妙：它把「拒绝」从一个死胡同，变成了一次纠偏。用户不必只是「拦下」，还能「拦下并告诉它该怎么改」——比如「别动 prod 配置，改 staging 那份」。模型收到这句反馈，就能调整后重来，而不是一头撞墙、不知所措。于是权限系统不只是一道「闸门」，更是一个把人的判断喂回给 agent 的回路：人在环，不仅是「卡住危险」，更是「把 agent 引向正确」。

always：会随信任而减少打扰的系统

always 这一档，是整套设计里最「人性化」的一笔。它做的不只是「放行这次」，而是把用户的这个决定 固化成一条规则、存进数据库（PermissionSaved → permission 表，带 project_id + action + resource 的唯一索引，按项目作用域）。下一次同样的「动作 × 资源」再来评估时，这条 saved 规则会让 evaluate 直接得到 allow——不再打扰用户。

用户答 always
对某 action×resource
saved.add →
写入 permission 表
project 作用域、跨会话存活
下次 evaluate →
命中该规则 → allow
静默放行、不再问

这条「always 会持久化」的设计，让整个权限系统有了一种会学习的体感：刚开始用，agent 干什么都问你，略显啰嗦；但你每点一次「以后都行」，就少了一类未来的打扰。用得越久、信任积累得越多，系统问得越少，越接近「你放心让它自动做的事，它就真的自动做了；你在意的事，它还是会先问」。这是一种动态平衡：初始保守（默认问），随用户授权逐步放开（always 攒规则），把「要多大自由度」的决定权，一点一点、可回收地交到用户手里——而不是一开始就让你在「全放开」和「事事问」之间做一个尴尬的二选一。源码里还有个体贴细节：当一条 always 规则被存下，它会顺手把其它正在挂起、恰好被这条新规则覆盖的询问也一并放行——你批准一次，相关的等待一起解除。

🗺️ 宏观图景

权限系统是贯穿整个工具系统的横切安全层，给「能力」装上「边界」：

  • 三态规则引擎（permission/schema.ts）：Rule = {action, resource, effect: allow|deny|ask}。evaluate 把各 ruleset 拼起、findLast 通配匹配（后匹配者胜），未命中默认 ask（安全默认：未知=交给人）。
  • assert 裁决流：工具 permission.assert → evaluate → deny 抛 DeniedError / allow 静默放行 / ask 发 permission.v2.asked 事件并挂起等回答（裹在 uninterruptibleMask 原子化）。
  • 三档回复 once/always/reject：once 仅此一次；always 放行 + 持久化规则（saved.add → SQLite permission 表、project 作用域）；reject → RejectedError（带反馈则 CorrectedError，把理由回传模型）。
  • 两级把关：deny 在「菜单层」（第 37 课 whollyDisabled，工具根本不出现给模型）；ask 在「调用层」（执行前发问）。纵深防御。
  • 会随信任减少打扰：默认保守（问），always 攒规则后逐步放开——动态平衡「放手干」与「人在环」。

有了权限这根缰绳，M7 工具系统的「能力 + 边界」就配齐了。下一课（第 42 课）回到另一条贯穿 M7 的红线——有界工具输出：一个 grep/bash 可能吐出几万行，怎么截断、怎么把全文外溢到托管文件，既不撑爆模型上下文、又不丢信息。第 43 课则讲 Skills 系统，M7 收尾。权限管「准不准做」，有界输出管「做完的结果怎么安全地交回给模型」——两者都是让一个强大 agent 真正「可控、可信、可托付」的关键拼图。

🔬 源码细节

整套引擎的核心，就是这个十来行的 evaluate（简化自 permission.ts）：

// 后匹配者胜；一条都没匹配上 → 默认 ask
function evaluate(action, resource, ...rulesets) {
  return rulesets.flat()
    .findLast(rule =>
      Wildcard.match(action, rule.action) && Wildcard.match(resource, rule.resource))
    ?? { action, resource, effect: "ask" }   // ← 安全默认
}

三处细节见功力。其一，findLast 而非 find——后定义的规则覆盖先定义的，于是「宽规则打底 + 窄规则特例」这种自然的表达成立（saved 规则、agent 规则、配置规则拼在一起，顺序即优先级）。其二，那个 ?? { effect: "ask" } 的兜底——把「没有规则覆盖」这一最常见的情况，稳稳接到「问人」上，而不是悄悄放行。其三，Wildcard.match 让规则能用 * 通配——{action:"*", resource:"*", effect:"deny"} 就是一条「全拒」兜底（missingAgentPermissions 给缺权限的 agent 用的正是它）。十来行代码，承载了「默认安全、可叠加覆盖、可通配」三个性质——好的安全机制，往往不靠复杂，而靠把对的默认值放在对的位置。

  • assert 裁决：deny→DeniedError（且第 37 课 whollyDisabled 让它不进菜单）；allow→静默放行；ask→发 permission.v2.asked 事件、uninterruptibleMask 内挂起等回答。

Over five lessons we equipped the agent with a powerful pair of hands: changing files, running commands, searching the web, fetching pages, asking people. But the greater the power, the more it needs a leash—an agent that can rm -rf, push code live, send outbound requests, if unmanaged, is a disaster not an assistant. This lesson's permission system is that leash. It answers a question running through all tools: this step—is it actually allowed? You may have noticed every prior tool's execute hides a permission.assert(...)—they all lead here. Tools cover "what can be done," permissions cover "what's allowed"; the former is capability, the latter the boundary, and together they make an agent both useful and controllable.

This system's core is surprisingly clean—it's essentially an allow / deny / ask three-state rule engine: each rule says "for some action on some resource, permit / deny / ask"; evaluation is last-match-wins, and when no rule matches, the default is "ask." This "default-ask" design is the soul of the whole security model: for the agent to do anything not pre-approved and sensitive, it must either hit an allow rule or stop and await the user's nod. Better still, the user's reply has three levels—once / always (and remember it) / reject—where always persists the decision into a rule, so the system asks less and less as trust accumulates. Grasp this lesson and you'll understand how opencode finds the dynamic balance point between "let the agent run free" and "keep a human in the loop."

🛂 Analogy
Picture the permission system as a building's security desk + a rulebook. Every time a worker (the agent) wants to enter a room to do something sensitive (run some tool on some resource), they badge at the desk. The desk flips the rulebook: marked "allow" → wave through; marked "deny" → blocked (and such rooms don't even appear on the worker's floor map); not mentioned in the rulebook → the desk buzzes the manager (user) to decide. The manager can say three things: "just this once" (let this through), "always" (let through, and the desk writes it into the rulebook so it won't bother the manager next time), "no" (blocked, possibly with a "why not" passed to the worker). Over time, the rulebook fills with the manager's standing decisions, and the desk buzzes less and less—the worker grows handier while the manager keeps final say over "new situations not in the book." Run free, but not out of control—that's the security desk's wisdom.

The three-state rule engine: allow / deny / ask

The permission system's core is a strikingly humble data structure. Open permission/schema.ts, and a rule is just three fields:

// permission/schema.ts —— one rule
Rule = { action: string, resource: string, effect: "allow" | "deny" | "ask" }
Ruleset = Rule[]

action (do what, e.g. edit/bash), resource (on whom, e.g. a file path), effect (what to do). The three effects are the system's entire "stance":

allow
permit, execute silently, don't bother the user
deny
refuse, throw DeniedError; and such a tool is kept off the menu by lesson 37's "whollyDisabled"
ask
ask the user—emit a permission.v2.asked event, stop and await an answer

Evaluation relies on an evaluate(action, resource, ...rulesets) function, just two lines of logic, every word crucial: concatenate the rulesets, findLast the last rule whose action and resource both wildcard-match, use its effect; if none match, default to effect: "ask". These two lines define the whole security model's two iron laws:

Last match wins (findLast)

Rules in order, later takes precedence. You can write a broad rule then override with a specific one—same approach as a firewall ruleset.

Default is "ask"

An action no rule mentions defaults to neither allow nor deny, but ask the human. This is "secure by default": unknown = leave it to a person.

This "default-ask," seemingly ordinary, is actually the design's keystone. It means: the agent can't quietly accomplish something unapproved by "gaming the rules"—any sensitive operation the rulebook didn't explicitly permit triggers a user prompt. New tools, new scenarios, edge cases… everything "not pre-considered" falls into the "ask" safety net. Defaulting "unknown" to "ask a human" rather than "allow" is this permission system's single most important choice. Reverse it to see the weight: if the default were "allow," then every new action the designers didn't foresee could let the agent silently do something you'd have wanted to block; defaulting to "ask" plugs this "slip-through" risk at the root—at the mere cost of a few extra prompts early on. Between "safety" and "convenience," this system unhesitatingly picks the former, then earns "convenience" back bit by bit via always.

Evaluation flow: one complete adjudication by assert

How does this ruleset act on a concrete tool call? Look at permission.assert—that line in every prior tool's execute. It launches a complete adjudication:

tool calls assert
{action, resource}
evaluate →
check effective ruleset
findLast, unmatched→ask
three-way branch →
allow→proceed
deny→DeniedError
ask→prompt, await

Three branches: deny → throw DeniedError outright, the tool can't run at all; allow → return silently, the tool runs; ask → emit a permission.v2.asked event and suspend (a Deferred pauses this execution in place) until the user answers. Note the whole assert is wrapped in uninterruptibleMask—the "ask—await—answer" sequence can't be interrupted midway, else you'd get "asked halfway, state scrambled." Again lesson 8's "atomize critical timing."

Worth pausing here to see that deny and ask actually guard two different gates—a kind of defense in depth:

deny · menu layer (lesson 37)

A wholly-denied tool is filtered out by whollyDisabled at materialize, not even appearing in the list sent to the model—the model doesn't even know "this tool exists."

ask · call layer (this lesson)

The tool is on the menu, the model can pick it, but each time it's actually about to execute, assert asks first—the last gate before execution.

The two levels' division is careful: "never allowed" things are made invisible to the model at the source (denying it even the chance to "think about it," sparing the explain-and-refuse round trips); "depends" things stay on the menu, asked at the last moment. The cleanest refusal is to not let the other know the option exists (echoing lesson 37), the most flexible gatekeeping is asking consent on the spot—deny and ask each do their job, one governing "can it exist," the other "is this time OK."

The user's reply has three levels, each corresponding to a different "future":

oncepermit just this once; the same thing next time still asks
alwayspermit + write into a persisted rule (saved.add → SQLite, project-scoped); future same thing is allowed directly, no more asking
rejectrefuse this time → RejectedError; with feedback → CorrectedError (carry "why not" back to the model)

The two forms of reject deserve a closer look. Just saying "no" (RejectedError) is pure interception; but if the user attaches a reason when rejecting, it becomes CorrectedError—and that reason flows back to the model with the error. This stroke is clever: it turns "rejection" from a dead end into a course-correction. The user needn't just "block" but can "block and tell it how to fix"—e.g. "don't touch the prod config, change the staging one." Receiving this feedback, the model can adjust and retry, rather than slamming into a wall, at a loss. So the permission system isn't just a "gate" but a loop feeding human judgment back to the agent: a human in the loop isn't only "stop the dangerous" but "steer the agent toward right."

always: a system that bothers you less as trust grows

The always level is the design's most "humane" stroke. It does more than "permit this time"; it solidifies the user's decision into a rule, stored in the database (PermissionSaved → the permission table, with a unique index on project_id + action + resource, project-scoped). Next time the same "action × resource" comes up for evaluation, this saved rule makes evaluate return allow directly—no more bothering the user.

user answers always
for some action×resource
saved.add →
write to permission table
project-scoped, survives across sessions
next evaluate →
match this rule → allow
silently permit, no more asking

This "always persists" design gives the whole permission system a learning feel: at first, the agent asks you about everything, a bit naggy; but every time you click "always," you shed a class of future interruptions. The longer you use it, the more trust accumulates, the less the system asks, approaching "what you're comfortable letting it do automatically, it really does automatically; what you care about, it still asks first." It's a dynamic balance: conservative at first (default-ask), gradually loosening with user authorization (always accrues rules), handing the decision of "how much freedom" to the user bit by bit, revocably—rather than forcing an awkward up-front choice between "fully open" and "ask about everything." The source has a thoughtful detail too: when an always rule is saved, it incidentally permits other pending asks that the new rule now covers—you approve once, and related waits all release together.

🗺️ The Big Picture

The permission system is the cross-cutting security layer through the whole tool system, fitting "capability" with a "boundary":

  • Three-state rule engine (permission/schema.ts): Rule = {action, resource, effect: allow|deny|ask}. evaluate concatenates rulesets, findLast wildcard-match (last match wins), unmatched defaults to ask (secure default: unknown = leave to a human).
  • assert adjudication: tool permission.assert → evaluate → deny throws DeniedError / allow proceeds silently / ask emits permission.v2.asked and suspends awaiting (wrapped in uninterruptibleMask for atomicity).
  • Three replies once/always/reject: once just this time; always permits + persists a rule (saved.add → SQLite permission table, project-scoped); reject → RejectedError (with feedback → CorrectedError, the reason back to the model).
  • Two gating levels: deny at the "menu layer" (lesson 37 whollyDisabled, the tool not even shown to the model); ask at the "call layer" (prompt before executing). Defense in depth.
  • Bothers less as trust grows: conservative by default (ask), gradually loosens after always accrues rules—dynamically balancing "run free" and "human in the loop."

With permissions as the leash, M7's tool system has its "capability + boundary" complete. Lesson 43 then covers the Skills system, wrapping up M7. Permissions govern "may it be done," bounded output governs "how the finished result is safely handed back to the model"—both key pieces making a powerful agent truly controllable, trustworthy, and worth entrusting.

🔬 Source Detail

The whole engine's core is this dozen-line evaluate (simplified from permission.ts):

// last match wins; if none match → default ask
function evaluate(action, resource, ...rulesets) {
  return rulesets.flat()
    .findLast(rule =>
      Wildcard.match(action, rule.action) && Wildcard.match(resource, rule.resource))
    ?? { action, resource, effect: "ask" }   // ← secure default
}

Three details show craft. One, findLast not find—later-defined rules override earlier, so the natural expression "broad rule as base + narrow rule as exception" holds (saved rules, agent rules, config rules concatenated, order = precedence). Two, that ?? { effect: "ask" } fallback—catching "no rule covers it," the most common case, squarely onto "ask a human," not silently allowing. Three, Wildcard.match lets rules use *—{action:"*", resource:"*", effect:"deny"} is a "deny-all" fallback (missingAgentPermissions for agents lacking permission is exactly this). A dozen lines carry "secure default, stackable override, wildcardable" three properties—a good security mechanism often relies not on complexity but on putting the right default in the right place.

🎯 Key Takeaways
  • Permission is the cross-cutting layer through all tools (core/src/permission.ts + permission/{schema,saved,sql}.ts): every prior tool's permission.assert leads here. Tools = "what can be done," permissions = "what's allowed."
  • Three-state rule engine: Rule={action, resource, effect: allow|deny|ask}. evaluate = concatenate rulesets then findLast wildcard-match (last match wins, order = precedence), unmatched defaults to ask (secure default: unknown left to a human, the agent can't game it into doing unapproved things silently).
  • assert adjudication: deny→DeniedError (and lesson 37's whollyDisabled keeps it off the menu); allow→proceed silently; ask→emit permission.v2.asked event, suspend awaiting within uninterruptibleMask.
  • Replies once/always/reject: once just this time; always→permit + persist a rule (saved.add→SQLite permission table, project-scoped, cross-session, next time allowed directly, and incidentally releasing other pending asks the new rule covers); reject→RejectedError/with-feedback CorrectedError (reason back to the model = steering, not just blocking).
  • Two gating levels + learning: deny at the menu layer, ask at the call layer (defense in depth); conservative by default (ask), gradually loosening after always accrues rules—dynamically balancing "run free" and "human in the loop," handing the freedom decision revocably to the user.
