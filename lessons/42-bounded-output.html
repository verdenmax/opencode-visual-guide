有界工具输出Bounded tool output

这一课收束一条从 M7 开篇就反复出现的红线——有界。前面我们一路见它：文件读的 offset/limit、搜索的 limit、bash 的 timeout 与字节上限。它们都在防同一件事：工具吐出来的结果，可能大到撑爆模型的上下文窗口。一个 grep import 可能命中几万行、一个 cat 大日志 能刷出几兆字节——若把这些原样塞回对话，瞬间填满模型那有限又昂贵的上下文，轻则花掉一大笔 token，重则把真正重要的对话内容挤出窗口。这一课的 ToolOutputStore，就是工具执行流上的最后一道有界闸门：它保证任何一个工具的输出，都不会洪水般淹没模型，又不让信息真的丢失。

它的解法有两处特别巧。其一，截断不是「砍掉尾巴」，而是「掐头去尾、留两端」——对日志、测试结果、文件转储这类输出，最有用的信息往往集中在开头（在跑什么）和结尾（结果/报错），中间是大段重复的躯干；所以它保留头一半 + 尾一半，中间用一个标记一笔带过。其二，截断的同时，把全文外溢（spill）到一份托管文件，并把文件路径一并交给模型——模型看到的是「精炼预览 + 全文在哪」，真要看全的，它可以用 read 工具去读那份文件。有界的视图、完整的备份、按需的访问——三者一配，既护住了上下文，又没真丢一个字节。读懂这一课，你就明白一个 agent 是怎么和「无限可能很大的工具输出」安全共处的。

📄 生活类比
想象一个工具跑完，吐出一份五百页的报告；可你（模型）手上一次只拿得下几页（上下文窗口就这么大）。怎么办？把五百页全塞给你——你一页都拿不住，全散了；只给你头五页——你又看不到最后的结论和报错，而那往往才是关键。ToolOutputStore 像一个机灵的助理：它给你开头几页 + 结尾几页（来龙与去果，通常最要紧），中间夹一张纸条「…中间省略，全文已归档在 42 号抽屉…」。你当下就抓住了大意；万一真要看中间，再去开那个抽屉（用 read 工具读那份托管文件）。大意先给你、全文存好、随时可取——既没让你被五百页淹没，又一个字都没丢。这，就是「有界但不失真」的助理之道。

为什么必须有界：上下文是稀缺资源

先想清楚动机。模型的上下文窗口是有限的——它一次能「读到」的内容有个上限（几万到几十万 token 不等），而且每个 token 都要花钱、都占时间。工具的输出却可能大得没边：

grep 太宽
搜一个常见词，命中几千上万行
cat 大文件
一个日志/数据文件，刷出几兆字节
bash 啰嗦
一条构建/测试命令，吐出海量进度与堆栈

如果把这些原样灌回对话，后果是双重的：一是烧钱又拖慢（几万行 token 的输入费用与延迟），二更致命——它会把真正重要的东西挤出窗口。上下文就那么大，被一坨无关的日志填满，模型就「记不住」前面的需求、刚才的计划了。所以「有界」不是小气，而是对一种稀缺资源的必要保护。回忆第 37 课：注册表的 settle 在工具执行后调了一句 resources.bound(...)——那个 resources 就是这一课的 ToolOutputStore，它是每个工具结果回到模型之前，都要过的最后一道关。

这里要和第 39 课的 bash 字节上限区分清楚——它们是两层不同的界，各防各的：

bash 的 1 MB 内存上限（第 39 课）

防的是「内存」：一条命令的 stdout 不能在进程里无限堆积、撑爆内存。是更早、更底层的一道闸。

ToolOutputStore 的界（本课）

防的是「上下文」：任何工具的结果回模型前，限到 2000 行 / 50 KB。是最终、面向模型的一道闸。

掐头去尾：截断的智慧

核心是 bound 函数和它调用的 preview。先看它认的几个界：

MAX_LINES = 2000
行数上限；可经 config 的 tool_output.max_lines 覆盖
MAX_BYTES = 50 KB
字节上限；可经 max_bytes 覆盖。行/字节双重把关
RETENTION = 7 天
托管文件保留期，过期自动清理

默认的界是 MAX_LINES = 2000 行、MAX_BYTES = 50 KB。当输出超界，preview 的截断策略很讲究——不是简单砍掉超出的部分，而是头尾各留一半：

超长输出
比如 1 万行
preview →
头 1000 行
headLines=⌈max/2⌉
+ 标记 +
尾 1000 行
tailLines=⌊max/2⌋

为什么是头 + 尾，而不是「头 N 行」这种最直觉的做法？因为对工具输出的信息分布有洞察：一段日志、一次测试运行、一个文件转储，最有价值的信息几乎总是落在两端——开头告诉你「在跑什么、什么配置」，结尾告诉你「结果如何、有没有报错、最终汇总」；中间往往是大段重复的进度行、数据行，信息密度最低。若天真地「只留前 1000 行」，恰恰会把最关键的『最终结果 / 报错』那一截砍掉——模型看了半天进度，却不知道到底成没成。preview 保留头一半、尾一半，中间夹一个 marker（如「…已截断…」），正是用同样的预算，留住信息密度最高的两端。字节超界时同理——头 headBytes、尾 tailBytes 各取一半。一个看似不起眼的「头尾都留」，背后是对『工具输出长什么样』的真实理解。不妨想想你自己看一份很长的测试日志：是不是也总先扫一眼开头「跑了哪些用例」，再直接跳到结尾看「几个通过几个失败、错在哪」？中间那几千行逐条进度，多半一眼带过。preview 的「掐头去尾」，做的正是这件最符合人直觉的事——它没有把截断当成一道无脑的「砍到 N 行」的机械操作，而是当成一个「在有限预算里，尽量多留有用信息」的优化问题来解。

外溢与回取：全文不丢，按需可读

截断必然损失信息——那被砍掉的中间一大段呢？这就是设计的第二处巧妙：截断给模型看的同时，把完整输出写进一份托管文件。write 把全文落到 {全局数据目录}/tool-output/tool_{递增ID}（用 flag: "wx" 独占创建、防覆盖），bound 把这个文件路径放进返回的 outputPaths。于是模型拿到的，是「预览 + 一条『全文在这个路径』的线索」：

完整输出
超界则 →
① 头尾预览给模型
+ marker
同时 →
② 全文 spill 到托管文件
outputPaths 交给模型

这一步把「有界」和「不失真」这对看似矛盾的目标，优雅地调和了。模型当下读到的是精炼预览，不会被淹没；可万一它真需要看被截掉的中间部分（比如要找某一行特定的报错），它手里有路径——用第 38 课的 read 工具去读那份文件就行（read 还能 offset/limit 翻页，正好对付大文件）。把「模型看到的」和「实际存下的」并排，这套分治就一目了然：

模型看到的

头尾预览（约 2000 行 / 50 KB 以内）+ 中间一个 marker + 一条 outputPaths 路径线索。够小，塞进上下文不心疼。

实际存下的

一份完整无损的托管文件（全文，无论多大）。模型按需用 read 去取，要哪段读哪段。

这是个漂亮的闭环：spill 出去的全文，靠 read 工具收得回来。整个工具系统在这里自我咬合——一个工具（任意）的超大输出，被另一个工具（read）兜底。这种「工具之间互相托底」的设计很值得回味：它没有给 ToolOutputStore 再造一套专门的「读回全文」接口，而是复用了早已存在的 read 工具——全文落成一份普通文件，而读文件这件事，工具系统里本就有人管。少造一个轮子，又让两块零件严丝合缝地咬上，正是第 36 课「一张统一的工具表」带来的红利：因为所有工具同形，它们才能这样互相衔接、彼此兜底。而这些托管文件也不会永远堆着：RETENTION = 7 天，过期自动清理，不让磁盘无限膨胀。有界的视图、完整的备份、按需的回取、自动的清理——一套完整的「大输出生命周期管理」。

🗺️ 宏观图景

有界工具输出，是工具结果回到模型前的最后一道闸，护住模型那稀缺的上下文：

  • 动机：上下文稀缺。工具输出可能极大（grep 万行、cat 几兆）；原样灌回会烧钱、拖慢，更会把真正重要的内容挤出窗口。第 37 课 settle 里的 resources.bound(...) 就是这一课。

  • 外溢 + 回取：截断的同时把全文 write 到托管文件（tool-output/tool_{递增ID}，flag:"wx" 独占），路径放进 outputPaths 交给模型；模型要看全的，用第 38 课 read 工具去读——有界视图 + 完整备份 + 按需访问，工具系统自我兜底。

到这里，M7 工具系统只剩最后一块拼图。第 43 课讲 Skills 系统：一种把「一套领域知识 + 一组操作步骤」打包、按需注入、还经权限化的特殊能力——它把前面所有零件（工具定义、注册、权限、Context Source）串成一个更高层的抽象。讲完它，M7「agent 的手」就彻底讲透：能定义工具、能收集调度、能读写搜跑问、能被权限管住、能把大输出收得住，最后还能把成套技能打包复用——一套完整、可控、可扩展的工具体系。

🔬 源码细节

头尾截断的核心，是 preview 里这段「各取一半」（简化自 tool-output-store.ts）：

// 超过 maxLines：保留头一半 + 尾一半，中间靠 marker 兜住
const headLines = Math.ceil(maxLines / 2)   // 头多拿一行（奇数时）
const tailLines = Math.floor(maxLines / 2)
const sampled = lines.length <= maxLines
  ? text
  : [ lines.slice(0, headLines).join("\n"),
      lines.slice(lines.length - tailLines).join("\n") ].join("\n")
// 字节也超界时：takePrefix(headBytes) + takeSuffix(tailBytes)，同样两端取

有两处工程上的细心。其一，headLines = ceil、tailLines = floor——当 maxLines 是奇数时，头比尾多留一行，一个微小却体现倾向的选择（开头通常略比结尾更值得多留一点上下文）。其二，boundedPreview 在拼装时，会把 marker 自身的字节数也算进预算（maxBytes - markerBytes - 4）——它不会「先按满额截断、再贴上 marker」而导致总量悄悄超界，而是预留出 marker 的位置再去截断。这种「连提示语本身占的空间都算进有界预算」的较真，正是「有界」二字的严格落地：说好了不超 50 KB，那连那句「…已截断…」也得算在 50 KB 之内。有界，是一个不容许『差不多』的承诺，连一句省略提示都得守约。

🎯 本课要点
  • 动机：模型上下文有限且昂贵，工具输出可能极大（grep 万行/cat 几兆），原样回灌会烧钱、拖慢、更会把重要内容挤出窗口。ToolOutputStore（core/src/tool-output-store.ts）是工具结果回模型前的最后一道有界闸（第 37 课 settle 调的 resources.bound）。
  • 掐头去尾截断：默认 MAX_LINES=2000 / MAX_BYTES=50KB（config/tool-output.ts 的 max_lines/max_bytes 可覆盖）；超界保留头一半（⌈/2⌉）+ 尾一半（⌊/2⌋）、中夹 marker——因信息密度集中两端（头=在跑啥、尾=结果/报错），只留前 N 行会砍掉最关键的结果。字节超界同理头尾各取一半。
  • 外溢 spill + 回取：截断同时把全文 write 到托管文件（{global.data}/tool-output/tool_{递增ID}，flag:"wx" 独占创建），路径入 outputPaths 交模型；要看全文用第 38 课 read 工具读（可翻页）——有界视图 + 完整备份 + 按需访问，工具系统自我兜底。
  • 两层界：bash 1 MB 内存上限（第 39 课）防内存 vs ToolOutputStore 2000 行/50 KB 防上下文。RETENTION=7 天 自动清理托管文件。
  • 严格的有界：boundedPreview 把 marker 自身字节也算进预算（maxBytes - markerBytes - 4），不让提示语本身使总量超界；headLines 用 ceil（奇数时头比尾多留一行）。「有界」是不容『差不多』的承诺。

This lesson ties off a red thread recurring since M7's start—bounding. We've seen it all along: file-read's offset/limit, search's limit, bash's timeout and byte cap. They all guard the same thing: a tool's output can be large enough to blow the model's context window. A grep import may hit tens of thousands of lines, a cat big-log can dump megabytes—stuffing these raw back into the conversation would instantly fill the model's limited, expensive context, at best burning a fortune in tokens, at worst pushing the genuinely important conversation out of the window. This lesson's ToolOutputStore is the final bounding gate on the tool-execution flow: it ensures no single tool's output can flood the model, without truly losing information.

Its solution has two clever parts. One, truncation isn't "chop off the tail" but "keep both ends, cut the middle"—for logs, test results, file dumps, the most useful info concentrates at the start (what's running) and the end (result/error), the middle being a long repetitive body; so it keeps the first half + last half, glossing the middle with a marker. Two, while truncating, it spills the full text to a managed file and hands the file path to the model too—the model sees "a crisp preview + where the full text is," and if it really needs the whole thing, it reads that file with the read tool. A bounded view, a complete backup, on-demand access—all three together protect the context yet lose not a byte. Grasp this lesson and you'll understand how an agent safely coexists with "tool output that could be infinitely large."

📄 Analogy
Imagine a tool finishes and spits out a five-hundred-page report; but you (the model) can only hold a few pages at once (the context window is just that big). What to do? Hand you all 500—you can't hold a single page, they all scatter; give you just the first five—you miss the final conclusion and errors, which are often the key. ToolOutputStore is like a clever assistant: it gives you the first few pages + the last few pages (the lead-up and the outcome, usually most important), with a slip in between: "…middle omitted, full report filed in drawer #42…." You grasp the gist right away; should you really need the middle, go open that drawer (read the managed file with the read tool). Gist up front, full text stored, retrievable anytime—you're not drowned by 500 pages, yet not a word is lost. This is the assistant's way of "bounded but not distorted."

Why it must be bounded: context is a scarce resource

First, the motivation. The model's context window is finite—what it can "read" at once has a ceiling (tens of thousands to hundreds of thousands of tokens), and every token costs money and time. Yet a tool's output can be boundlessly large:

grep too broad
searching a common word, hitting thousands of lines
cat a big file
a log/data file dumping megabytes
bash verbose
a build/test command spitting masses of progress and stacks

Pouring these raw back into the conversation has a double consequence: one, burning money and slowing things (tens of thousands of lines' token cost and latency), two, more fatal—it pushes the genuinely important out of the window. The context is only so big; filled with a lump of irrelevant logs, the model "forgets" the earlier requirements, the just-made plan. So "bounding" isn't stinginess but necessary protection of a scarce resource. Recall lesson 37: the registry's settle calls resources.bound(...) after a tool executes—that resources is this lesson's ToolOutputStore, the final gate every tool result passes before returning to the model.

Here we must distinguish it from lesson 39's bash byte cap—they're two different bounds, each guarding its own:

bash's 1 MB memory cap (lesson 39)

Guards "memory": a command's stdout can't pile up unboundedly in the process and blow memory. An earlier, lower-level gate.

ToolOutputStore's bound (this lesson)

Guards "context": any tool's result, before returning to the model, is bounded to 2000 lines / 50 KB. The final, model-facing gate.

Cut head and tail: the wisdom of truncation

The core is the bound function and the preview it calls. First the bounds it honors:

MAX_LINES = 2000
line cap; overridable via config's tool_output.max_lines
MAX_BYTES = 50 KB
byte cap; overridable via max_bytes. Lines/bytes dual gating
RETENTION = 7 days
managed-file retention, auto-cleaned on expiry

The default bounds are MAX_LINES = 2000 lines, MAX_BYTES = 50 KB. When output exceeds, preview's truncation strategy is careful—not simply chopping the excess, but keeping half at each end:

over-long output
say 10,000 lines
preview →
first 1000 lines
headLines=⌈max/2⌉
+ marker +
last 1000 lines
tailLines=⌊max/2⌋

Why head + tail, not the most intuitive "first N lines"? Because of insight into tool output's information distribution: a log, a test run, a file dump—the most valuable info almost always sits at the two ends—the start tells you "what's running, what config," the end tells you "the result, any error, the final summary"; the middle is often a long repetitive run of progress/data lines, lowest in info density. Naively "keep only the first 1000 lines" would precisely chop off the all-important 'final result / error'—the model reads progress forever yet doesn't know whether it succeeded. preview keeps the first half, last half, with a marker between (like "…truncated…"), exactly using the same budget to keep the highest-info-density two ends. Same when bytes exceed—head headBytes, tail tailBytes, half each. A seemingly trivial "keep both ends" rests on a real understanding of "what tool output looks like." Think of how you read a very long test log: don't you also first glance at the start "which cases ran," then jump straight to the end "how many passed/failed, where the error is"? The thousands of progress lines in between, mostly skimmed past. preview's "cut head and tail" does exactly this most-human-intuitive thing—it treats truncation not as a brainless "chop to N lines" mechanical op, but as an optimization problem of "keep as much useful info as possible within a limited budget."

Spill and retrieve: the full text isn't lost, readable on demand

Truncation inevitably loses info—what about that big chopped middle? This is the design's second cleverness: while truncating what the model sees, it writes the full output into a managed file. write lands the full text to {global data dir}/tool-output/tool_{ascending ID} (with flag: "wx" exclusive create, guarding against overwrite), and bound puts this file path into the returned outputPaths. So the model gets "a preview + a clue that 'the full text is at this path'":

full output
if over bound →
① head-tail preview to model
+ marker
simultaneously →
② full text spilled to managed file
outputPaths to the model

This step elegantly reconciles the seemingly contradictory goals of "bounded" and "undistorted." The model reads a crisp preview now, not drowned; but should it really need the chopped middle (e.g. to find one specific error line), it has the path—just read that file with lesson 38's read tool (read can offset/limit-page too, perfect for big files). Lay "what the model sees" and "what's actually stored" side by side and this divide-and-conquer is plain:

What the model sees

A head-tail preview (within ~2000 lines / 50 KB) + a marker in the middle + an outputPaths path clue. Small enough to fit the context painlessly.

What's actually stored

A complete, lossless managed file (full text, however large). The model fetches on demand with read, reading whatever section it needs.

It's a beautiful loop: the spilled-out full text is retrievable via the read tool. The whole tool system meshes with itself here—any tool's oversized output is backstopped by another tool (read). This "tools backstopping each other" design is worth savoring: it didn't build ToolOutputStore a dedicated "read back the full text" interface, but reused the already-existing read tool—the full text lands as an ordinary file, and reading files is something the tool system already handles. One fewer wheel reinvented, and two parts mesh seamlessly—exactly the dividend of lesson 36's "one unified tool form": because all tools are isomorphic, they can interconnect and backstop each other like this. And these managed files don't pile up forever: RETENTION = 7 days, auto-cleaned on expiry, not letting the disk grow unboundedly. A bounded view, a complete backup, on-demand retrieval, automatic cleanup—a complete "large-output lifecycle management."

🗺️ The Big Picture

Bounded tool output is the final gate before tool results return to the model, protecting the model's scarce context:

  • Motivation: context is scarce. Tool output can be huge (grep ten-thousand lines, cat megabytes); pouring it raw back burns money, slows things, and pushes the genuinely important out of the window. Lesson 37 settle's resources.bound(...) is this lesson.
  • Head-tail truncation (preview): default MAX_LINES=2000 / MAX_BYTES=50KB (config-overridable); over bound keeps the first half + last half, marker in between—because tool output's info density concentrates at both ends (start=what's running, end=result/error), "keep only first N lines" would chop the all-important result.
  • Spill + retrieve: while truncating, write the full text to a managed file (tool-output/tool_{ascending ID}, flag:"wx" exclusive), put the path in outputPaths for the model; to see the full thing, read it with lesson 38's read tool—bounded view + full backup + on-demand access, the tool system backstopping itself.
  • Distinguish the two bounds: bash's 1 MB memory cap (lesson 39) guards memory; ToolOutputStore's 2000-line/50 KB guards context. RETENTION=7 days auto-cleans managed files.

By here, M7's tool system has one last piece left. Lesson 43 covers the Skills system: a special capability packaging "a body of domain knowledge + a set of operating steps," injected on demand and permissioned—stringing all the prior parts (tool definition, registration, permissions, Context Source) into a higher-level abstraction. After it, M7's "the agent's hands" is fully covered: able to define tools, collect and dispatch them, read/write/search/run/ask, be governed by permissions, hold large outputs in check, and finally package whole skills for reuse—a complete, controllable, extensible tool system.

🔬 Source Detail

The head-tail truncation's core is this "half each" in preview (simplified from tool-output-store.ts):

// over maxLines: keep first half + last half, marker backstops the middle
const headLines = Math.ceil(maxLines / 2)   // head takes one extra (when odd)
const tailLines = Math.floor(maxLines / 2)
const sampled = lines.length <= maxLines
  ? text
  : [ lines.slice(0, headLines).join("\n"),
      lines.slice(lines.length - tailLines).join("\n") ].join("\n")
// when bytes also exceed: takePrefix(headBytes) + takeSuffix(tailBytes), both ends too

Two engineering niceties. One, headLines = ceil, tailLines = floor—when maxLines is odd, the head keeps one more line than the tail, a tiny but telling preference (the start usually merits slightly more context than the end). Two, boundedPreview, when assembling, counts the marker's own bytes into the budget (maxBytes - markerBytes - 4)—it won't "truncate to full quota then paste the marker" and quietly overshoot, but reserves the marker's room before truncating. This rigor of "even the notice's own space counts toward the budget" is the strict landing of "bounded": promised not to exceed 50 KB, so even that "…truncated…" must count within the 50 KB. Bounded is a promise that allows no "close enough."
