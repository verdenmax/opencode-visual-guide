Skills 系统The skills system

M7 一路走来，我们把 agent 的「手」拆得很细：怎么定义一个工具（L36）、怎么收集调度（L37）、文件/搜索/执行/网络各类工具（L38~40）、权限（L41）、有界输出（L42）。这最后一课的 Skills 系统，是把前面所有零件串成一个更高层抽象的压轴戏——它回答一个很实际的问题：怎么给 agent「教」一套新本领，而不用重训模型、也不用往代码里硬塞逻辑？答案是「技能（skill）」：把「一套领域知识 + 一组操作步骤 + 配套的脚本/文件」打包成一个单元，需要时按需加载到对话里。想给 agent 加一项能力？往技能库里丢一个 skill 就行。

Skills 最精妙的，是它的两段式架构——这也是这一课的核心，恰好让 M5（System Context）和 M7（工具）在此交汇。一个 skill 被劈成两半：「名字」那半，常驻在系统上下文里（走 Context Source，第 21~27 课）——模型随时都看得见有哪些技能（只是名字 + 一句描述，极省 token）；「正文」那半，按需经一个权限化的 skill 工具加载（第 36/41 课）——只有当任务真的撞上某个技能时，模型才调用 skill 工具，把那套完整的指令与资源注入进来。这正是「渐进式披露 / 懒加载」：名字便宜地全摆出来，正文昂贵地按需才取。读懂这个两段式，你不仅懂了 Skills，更看见了整个 opencode 反复用的那个核心套路——先廉价地『广而告之』，再昂贵地『按需兑现』。

📚 生活类比
把技能库想象成办公室墙上一排贴了标签的活页夹，每个夹子里是一套标准作业流程（SOP）外加一叠配套表单和脚本。书架最显眼处挂着一张索引卡——只印每个夹子的书脊标签（技能名 + 一句话说明），始终在视野里（这就是系统上下文）。agent 接到活儿，先扫一眼索引卡；要是某个标签正好对上手头的任务，它就把那个夹子抽下来、翻开（调用 skill 工具）——此刻，完整的 SOP 加上「配套资料在哪个文件夹」就摊到了桌面上。接着 agent 照着 SOP 一步步做，用它平时那些工具（read 去读引用的脚本、bash 去跑）把活干完。给 agent 添一项新本领，不靠重新培训它，而靠往书架上多放一个活页夹。索引常驻、正文按需——这就是 Skills 的两段式智慧，也是一种「轻装上阵、用时再取」的从容。

两段式：名字常驻，正文按需

先看清一个 skill 到底由什么构成——它远不止「一段提示词」，而是一个自带资料的知识包：

name
技能名（模型调用时用它指名）
description
一句话说明（模型据此判断「这任务该不该用它」）
content
完整正文：详细的工作流指令
基准目录 + files
一个文件夹，装着配套脚本/参考资料；正文里可引用它们

Skills 系统的灵魂，是把这样一个技能劈成两半，分走两条完全不同的通道。这张对照表是这一课最该记住的东西：

「名字」半 · 走 Context Source

SkillGuidance.load(agent) 返回一个 SystemContext（第 21~27 课的源！），往系统上下文注入一份技能清单——每个只有 <name> + <description>，外加一句「任务匹配某技能时，用 skill 工具加载它」。常驻、便宜、模型随时看得见。

「正文」半 · 走 skill 工具

skill 工具（经内部 permission.assert(action: skill) 权限化，非 withPermission）：input 一个 {name}。模型调用它，toModelOutput 把该技能的完整正文（指令 + 基准目录 + 文件清单）注入对话。按需、昂贵、只在用到时取。

为什么非要劈成两半？因为「全都加载」根本行不通。设想技能库里有上百个技能，每个正文动辄几千字——若开局就把它们的全文一股脑塞进上下文，窗口瞬间爆满，正经对话没地方放了。但模型又必须知道有哪些技能可用，否则它根本想不到去调用。两段式正是这对矛盾的优雅解：把「有什么」（廉价的名字）和「具体是什么」（昂贵的正文）分开——名字便宜，全摆出来当目录；正文昂贵，谁用到谁再取。这和第 37 课注册表的「definitions 全列、settle 按需执行」、第 42 课的「预览常驻、全文 spill 待取」，是同一个套路在不同地方的第三次现身。opencode 反反复复在用这一招：先廉价地广而告之，再昂贵地按需兑现。

一次技能加载的完整流程

把两段式串成一条时间线，一个技能从「被看见」到「被用上」是这样走的：

①广告SkillGuidance 把技能名+描述注入系统上下文（Context Source），模型一直看得见
②匹配模型遇到一个任务，发现某技能的 description 正好对上
③调用模型调 skill 工具，传入那个 name（先过权限：第 41 课）
④注入toModelOutput 注入完整正文：指令 + 基准目录 URL + <skill_files> 清单
⑤执行模型照正文一步步做，用 read 读引用的脚本/文件、用 bash 跑——靠的是它平时那些工具

第 ④⑤ 步藏着 Skills 最漂亮的一点：skill 工具本身并不「干活」，它只「发讲义」。它注入的正文里，除了文字指令，还有一句关键的「本技能的基准目录在此 URL」，以及一份 <skill_files> 文件清单。也就是说，一个技能 = 「一份说明书 + 一个装着脚本和资料的文件夹」。skill 工具把说明书和文件夹位置交给模型，接下来真正的活儿，是模型用它已有的通用工具去干的——read 去读 scripts/ 下的脚本、bash 去执行、edit 去改文件。这是一种漂亮的分工：skill 负责「把对的剧本递到手上」，而执行剧本用的，还是 M7 前面那套通用的手。技能不扩展 agent 的「能做什么」，而扩展它的「知道该怎么做」。这个区分值得多想一层：模型本身已经会用 read/edit/bash 这些工具了（能做什么是固定的），它缺的常常是「面对某类具体任务，该按什么顺序、用什么诀窍把这些工具组织起来」——而这，恰恰是一个 skill 的正文所承载的。换句话说，skill 注入的不是新能力，而是一套把已有能力用对的「方法论」。这也解释了为什么 skill 适合打包「领域最佳实践」：把一个老手脑子里「这种活儿要先这样、再那样、注意别踩那个坑」的经验，写成一份正文，需要时贴给模型——agent 于是像是临时请到了一位带着 SOP 的专家。

🗺️ 宏观图景 · M7 收官

Skills 是 M7 工具系统的压轴，把前面所有零件拼成「给 agent 教新本领」的高层抽象：

  • skill = 知识 + 步骤 + 配套文件的打包单元，按需加载。给 agent 加能力 = 往技能库丢一个 skill，不必重训模型、不必改代码。
  • 两段式架构（核心）：名字半经 SkillGuidance → SystemContext（M5 Context Source）常驻注入（名+描述，省 token、随时可见）；正文半经权限化的 skill 工具按需注入（完整指令 + 基准目录 + 文件清单）。渐进式披露 / 懒加载。
  • skill 只「发讲义」、不「干活」：注入正文（含基准目录与 <skill_files>），真正执行靠模型已有的通用工具（read 读脚本、bash 跑、edit 改）。技能扩展的是「知道该怎么做」，不是「能做什么」。
  • M5+M7 的交汇点：Skills 用全了 Context Source（M5）+ Tool.make（L36）+ 注册表（L37）+ 权限（L41）+ read 等（L38）；半是上下文、半是工具。SkillDiscovery 可从目录索引、从 URL 下载——可发现、可分发、可扩展。

到这里，M7「agent 的手」彻底讲透：从「一个工具怎么定义」（L36），到收集调度（L37）、各类具体工具（L38~40）、权限边界（L41）、有界输出（L42），再到把成套本领打包复用（L43）。一套完整、可控、可扩展的工具体系，让 LLM 这颗「脑」真正能动手改变世界。下一个 part（M8）将转向配置、Agents 与 Provider——agent 是怎么被配置出来的、不同 agent 怎么定义、模型与供应商怎么接入。脑（M6 LLM）、手（M7 工具）都齐了，接下来该讲怎么把一个具体的 agent「攒」出来、配好它的脑与手、让它真正跑起来。

🔬 源码细节

skill 工具注入正文的 toModelOutput，把「说明书 + 文件夹」交得很清楚（简化自 tool/skill.ts）：

// 注入给模型的技能正文：指令 + 基准目录 + 文件清单
[ `<skill_content name="${skill.name}">`,
  `# Skill: ${skill.name}`,
  skill.content.trim(),                          // 完整指令正文
  `Base directory for this skill: ${dirURL}`,   // 配套文件在哪
  "Relative paths (e.g., scripts/) are relative to this base directory.",
  "<skill_files>", ...files.map(f => `<file>${f}</file>`), "</skill_files>",
].join("\n")

这段注入有两处用心。其一，它给的是基准目录的 URL + 一份文件清单，而非把那些脚本文件的内容也一并塞进来——又是「渐进式披露」：先告诉模型「有哪些文件、在哪」，至于某个脚本具体写了啥，等模型真要用时，再用 read 工具去读。一层套一层的懒加载：技能正文懒加载、技能里的文件再懒加载。其二，注释里那句「file list is sampled（文件清单是抽样的）」很诚实——当一个技能附带成百上千个文件时，清单本身也会被有界（呼应第 42 课），不让「列文件」这件事反过来撑爆上下文。从技能正文到文件清单，处处是『按需、有界』——这套贯穿 M7 的纪律，到收官这一课依然一以贯之。

🎯 本课要点
  • skill = 知识+步骤+配套文件的打包单元（core/src/skill/*、tool/skill.ts），按需加载。给 agent 加本领 = 往技能库丢一个 skill，不重训模型、不改代码。
  • 两段式架构（核心）：名字半经 SkillGuidance.load → SystemContext（M5 Context Source，第 21~27 课）常驻注入（仅 name+description，省 token、随时可见）；正文半经 skill 工具（Tool.make + 内部 permission.assert(action: skill)，非 withPermission；input {name}）按需注入完整正文。渐进式披露/懒加载——与第 37 课「definitions 全列/settle 按需」、第 42 课「预览常驻/全文 spill」同一套路。
  • skill 只「发讲义」、不「干活」：toModelOutput 注入 <skill_content>（指令 + 基准目录 URL + <skill_files> 清单）；真正执行靠模型已有通用工具（read 读脚本、bash 跑、edit 改）。技能扩展「知道该怎么做」，非「能做什么」。
  • M5+M7 交汇点：Skills 用全 Context Source（M5）+ Tool.make（L36）+ 注册表（L37，SkillTool.layer）+ 权限（L41）+ read（L38）。一只脚在上下文、一只脚在工具——半上下文、半工具。是建立在前面所有抽象之上的更高层复用。
  • 可发现/可分发：SkillDiscovery 从目录索引、能从 URL 下载技能。文件清单标注 sampled（抽样、有界），层层懒加载（技能正文懒加载、技能内文件再懒加载）——「按需、有界」纪律贯穿到 M7 收官。

Through M7 we've dissected the agent's "hands" finely: how to define a tool (L36), collect and dispatch (L37), file/search/exec/network tools (L38–40), permissions (L41), bounded output (L42). This final lesson's Skills system is the closing act that strings all the prior parts into a higher-level abstraction—it answers a very practical question: how do you "teach" an agent a new competence without retraining the model or hardcoding logic into the code? The answer is a "skill": packaging "a body of domain knowledge + a set of operating steps + accompanying scripts/files" into a unit, loaded on demand into the conversation. Want to add a capability to the agent? Just drop a skill into the skill library.

Skills' most exquisite part is its two-stage architecture—this lesson's core, where M5 (System Context) and M7 (tools) meet. A skill is split in two: the "name" half lives resident in the system context (via Context Source, lessons 21–27)—the model always sees which skills exist (just name + a one-line description, very token-cheap); the "body" half loads on demand via a permissioned skill tool (lessons 36/41)—only when a task truly matches a skill does the model call the skill tool to inject that full set of instructions and resources. This is "progressive disclosure / lazy loading": advertise the names cheaply, fetch the body expensively on demand. Grasp this two-stage design and you've understood not just Skills but the core pattern opencode uses again and again—cheaply "advertise" first, expensively "fulfill on demand" later.

📚 Analogy
Picture the skill library as a row of labeled binders on the office wall, each holding a standard operating procedure (SOP) plus a stack of accompanying forms and scripts. Most prominently on the shelf hangs an index card—printing only each binder's spine label (skill name + a one-line note), always in view (this is the system context). The agent, handed a task, first glances at the index card; if a label matches the task at hand, it pulls that binder down and opens it (calls the skill tool)—now the full SOP plus "which folder the supporting materials are in" is spread on the desk. Then the agent follows the SOP step by step, using its usual tools (read to read the referenced scripts, bash to run them) to finish the work. Adding a new competence to the agent isn't by retraining it, but by putting one more binder on the shelf. Index resident, body on demand—this is Skills' two-stage wisdom.

Two stages: names resident, body on demand

First see clearly what a skill consists of—it's far more than "a prompt," but a knowledge package with its own materials:

name
the skill name (the model uses it to call by name)
description
a one-line note (the model judges "should this task use it" by this)
content
the full body: detailed workflow instructions
base dir + files
a folder holding accompanying scripts/references; the body can reference them

The Skills system's soul is splitting such a skill in two, routed down two completely different channels. This table is the thing to remember most from this lesson:

"name" half · via Context Source

SkillGuidance.load(agent) returns a SystemContext (lessons 21–27's source!), injecting into the system context a skill catalog—each with just <name> + <description>, plus "when a task matches a skill, load it with the skill tool." Resident, cheap, always visible to the model.

"body" half · via the skill tool

The skill tool (permissioned via an internal permission.assert(action: skill), not withPermission): input a {name}. The model calls it, and toModelOutput injects that skill's full body (instructions + base directory + file list) into the conversation. On demand, expensive, fetched only when used.

Why split in two? Because "load everything" simply won't work. Imagine the library has hundreds of skills, each body thousands of words—loading all their full text into context at the start would instantly max out the window, leaving no room for the actual conversation. Yet the model must know which skills are available, else it'd never think to call one. The two stages elegantly resolve this tension: separate "what exists" (cheap names) from "what it specifically is" (expensive body)—names cheap, all laid out as a catalog; body expensive, fetched only by whoever needs it. This, lesson 37's registry "list all definitions, settle on demand," and lesson 42's "preview resident, full text spilled for retrieval" are the same pattern's third appearance in a different place. opencode uses this move over and over: cheaply advertise first, expensively fulfill on demand.

One skill load's full flow

String the two stages into a timeline, and a skill's journey from "seen" to "used" goes:

①advertiseSkillGuidance injects skill name+description into system context (Context Source), the model always sees it
②matchthe model meets a task, finds some skill's description fits exactly
③callthe model calls the skill tool with that name (passing permission first: lesson 41)
④injecttoModelOutput injects the full body: instructions + base directory URL + <skill_files> list
⑤executethe model follows the body step by step, using read for referenced scripts/files, bash to run—via its usual tools

Steps ④⑤ hide Skills' most beautiful point: the skill tool itself doesn't "do the work," it only "hands out the handbook." The body it injects, besides text instructions, has a key line "this skill's base directory is at this URL" and a <skill_files> file list. That is, a skill = "a manual + a folder holding scripts and materials." The skill tool hands the manual and folder location to the model, and the actual work that follows is done by the model using its existing general tools—read for the scripts under scripts/, bash to run, edit to change files. It's a beautiful division of labor: the skill handles "putting the right script into your hands," and executing the script still uses M7's earlier general hands. A skill doesn't extend the agent's "what it can do" but its "knowing how to do it." This distinction merits one more layer of thought: the model already knows how to use read/edit/bash (what it can do is fixed); what it often lacks is "facing a specific kind of task, in what order and with what tricks to organize these tools"—and that's exactly what a skill's body carries. In other words, a skill injects not new capability but a "methodology" for using existing capabilities right. This also explains why skills suit packaging "domain best practices": writing an old hand's "for this kind of job, first do this, then that, watch out for that pit" experience into a body, pasted to the model when needed—the agent thus seems to have temporarily hired an expert bearing an SOP.

🗺️ The Big Picture · M7 Finale

Skills is M7's tool-system finale, stitching all prior parts into a high-level "teach the agent a new competence" abstraction:

  • skill = a packaged unit of knowledge + steps + accompanying files, loaded on demand. Adding capability = drop a skill into the library, no retraining the model, no changing code.
  • Two-stage architecture (core): the name half resident-injected via SkillGuidance → SystemContext (M5 Context Source) (name+description, token-saving, always visible); the body half injected on demand via the permissioned skill tool (full instructions + base directory + file list). 🔬 Source Detail

The skill tool's toModelOutput, injecting the body, hands over "manual + folder" clearly (simplified from tool/skill.ts):

// the skill body injected to the model: instructions + base dir + file list
[ `<skill_content name="${skill.name}">`,
  `# Skill: ${skill.name}`,
  skill.content.trim(),                          // the full instruction body
  `Base directory for this skill: ${dirURL}`,   // where the supporting files are
  "Relative paths (e.g., scripts/) are relative to this base directory.",
  "<skill_files>", ...files.map(f => `<file>${f}</file>`), "</skill_files>",
].join("\n")

Two careful touches in this injection. One, it gives the base directory URL + a file list, not stuffing those script files' contents in too—again "progressive disclosure": first tell the model "which files exist, where," and as for what a script specifically says, when the model really needs it, it reads it with the read tool. Lazy loading nested in lazy loading: the skill body lazy-loaded, the files within the skill lazy-loaded again. Two, the comment's "file list is sampled" is honest—when a skill comes with hundreds or thousands of files, the list itself is bounded (echoing lesson 42), not letting "listing files" itself blow the context. From the skill body to the file list, "on demand, bounded" is everywhere—this discipline running through M7 holds firm even in this closing lesson.
