Account 与设备码 OAuthAccount & device-code OAuth

你在终端里第一次让 opencode 连上某个账号/服务，会看到这样一幕：屏幕上显示一个短码（如 WDJB-MJHT）和一个网址，让你「去浏览器打开这个网址、输入这个码」；你照做、点了授权，回到终端，它已经自动登录好了。这背后是 设备码（device-code）OAuth 流——一种专为「没有浏览器回调」的 CLI/终端设计的登录方式（opencode/src/account/account.ts）。拿到的凭据则进了一个统一的凭据库（core/src/credential.ts），之后被喂给 provider 去调模型。这一课就讲清楚：CLI 没有浏览器回调，怎么完成 OAuth？拿到的密钥/令牌，又是怎么存、怎么喂给 provider 的？这是一块横切整个运行时、却在前面 64 课里始终没正面讲过的拼图——你用 opencode 连模型、连服务，背后总得先「登录、拿到凭据」，而这一课正是补上「身份与凭据」这一环。

这一课最值得带走的洞见有两个。第一，设备码流把「在哪授权」和「令牌落到哪」解耦了：传统网页 OAuth 靠「浏览器回调」把授权结果送回应用——但 CLI 是个终端进程，没有浏览器、收不了回调。设备码的巧思是：终端只显示一个短码，你在任何一台有浏览器的设备上（手机、另一台电脑）打开网址、输码、授权；与此同时，终端在后台「轮询」令牌端点，直到你那边授权完成，它就拿到了令牌。「授权」发生在浏览器、「令牌」落到 CLI，两端用一个短码牵线。第二，凭据被统一成一个带类型标签的「值」：无论是 OAuth 令牌（access+refresh+expires）还是直接的 API Key，都被塑成同一个 Credential.Info 标签联合，进同一个有 CRUD 接口的凭据库——又一次「异质塑同形」（同 L66/L36），让上层「取个凭据喂给 provider」时不必关心它是哪种。把这两点合起来，这一课其实在讲一个完整的故事：opencode 作为一个要连各种外部服务（模型 provider、MCP、集成）的工具，必须解决「怎么让用户登录、怎么把拿到的密钥安全地存好、再可靠地用上」——而它给出的答案，是一套「设备码登录 + 统一凭据库」的组合拳。

🎫 生活类比
设备码流，像极了取餐叫号。你在一家忙碌的餐厅点餐（CLI 发起登录），柜台给你一张写着取餐号的小票（user_code + 那个网址）——但你不用傻站在柜台等（CLI 没法「收回调」）。你可以拿着号去任何地方：去自助机上输号确认、找经理签字（在任意一台有浏览器的设备上打开网址、输码、授权）。与此同时，柜台的叫号屏在后台反复刷：「还没好……还没好……（authorization_pending）」「人太多、慢点刷（slow_down）」，直到你那边一确认，屏幕「叮」地跳出你的餐（access_token）。妙处就在：「确认」在别处发生、「取餐」在柜台完成，一张号把两头牵起来——这恰好绕过了「柜台没法主动找到你」（CLI 收不了回调）这个死结。而拿到的「餐」（凭据），无论是一张 OAuth 月票（会过期、能续）还是一把一次性钥匙（API Key），都被装进同一种储物柜（凭据库），凭一个柜号（id）随时取用；月票快到期了，店家还会提前 5 分钟悄悄给你续上（eager refresh），你压根不会被「票过期了」打断。

设备码流：CLI 没有浏览器回调，怎么 OAuth

网页应用做 OAuth，靠的是浏览器回调（redirect）：用户在授权页点了同意，授权服务器把结果重定向回应用预先登记的回调地址。但 CLI/终端是个没有浏览器、没有公网回调地址的进程——这条路走不通。你不可能让一个跑在别人电脑终端里的程序，去登记一个「授权完跳回来」的网址（它没有网址，也没有浏览器接住跳转）。设备码授权（OAuth 2.0 Device Authorization Grant）就是为这种「输入/显示受限」的设备设计的标准方案——它最早是给智能电视、游戏机这类「不方便输密码、没浏览器」的设备用的，而 CLI 恰好是同一类困境。它的巧思是把流程拆成「显示码 + 后台轮询」两半，让「需要浏览器的那一步」挪到用户手边任意一台方便的设备上去做：

网页 OAuth（CLI 用不了）

跳转到授权页 → 用户同意 → 浏览器回调带 code 回应用。依赖「应用能收到浏览器回调」——CLI 没有这个。

设备码流（CLI 用这个）

终端显示 user_code+URL → 用户在别处授权 → 终端轮询令牌端点直到成功。不需要回调，只需「显示 + 轮询」。

源码把每一步都建了模。第一步换取 DeviceAuth：{device_code（保密、轮询用）, user_code（给人看、人去输）, verification_uri_complete（授权网址）, expires_in（多久过期）, interval（多久轮询一次）}。注意这里两个码的分工很讲究：user_code 是给人看的、短而好念（方便你在另一台设备上手输），device_code 是给机器用的、保密的（CLI 揣着它去换令牌，绝不显示给人）。终端把 user_code 和网址显示给你，自己揣着 device_code。第二步是轮询：终端每隔 interval 拿 device_code 去问令牌端点（DeviceTokenRequest{grant_type, device_code, client_id}，client_id="opencode-cli"），返回要么是 DeviceTokenSuccess{access_token, refresh_token, token_type:"Bearer", expires_in}，要么是 DeviceTokenError——而 DeviceTokenError.toPollResult() 把 OAuth 的错误字符串翻译成清晰的轮询状态，这是整个轮询循环的「红绿灯」。这个 toPollResult 是一处很值得品的小设计：OAuth 协议里，「还没授权」「轮太快了」「过期了」「被拒了」全都以 error 字符串的形式回来，混在一起；toPollResult 把这些字符串翻成有名字的状态类（PollPending/PollSlow/…），于是轮询循环的代码就能干净地 switch 这些状态、而不是到处比对魔法字符串——又一次「把字符串/原始值升格成有类型的值」（同 L20 RunError、L05 错误即值）：

error 字符串→ 轮询状态含义 / 该怎么办
authorization_pendingPollPending用户还没授权完——继续等、接着轮询
slow_downPollSlow轮太勤了——把间隔放慢
expired_tokenPollExpireddevice_code 过期了——重新来过
access_deniedPollDenied用户拒绝了——结束

把这条「申请→显示→（你去别处授权）→轮询→拿令牌」的链路走一遍，你就看清了设备码的全貌：

1 CLI 申请，拿回 DeviceAuth（device_code + user_code + 网址 + interval）
2 终端显示：「打开 <网址>，输入 user_code」
3 你在手机/浏览器打开网址、输码、点授权（这一步在 CLI 之外）
4 同时 CLI 每隔 interval 轮询：Pending…Pending…（你授权后）→ DeviceTokenSuccess
5 拿到 access_token+refresh_token，登录完成；快过期前 5 分钟（eagerRefreshThreshold）自动续

凭据库：OAuth 令牌与 API Key，一个储物柜

登录拿到的东西要存起来、之后反复取用，这是 core/src/credential.ts 的活。为什么需要一个专门的「凭据库」，而不是把令牌随手塞个变量？因为凭据有它自己的一套生命周期与多样性：它有多种形态（OAuth 令牌 / API Key）、要长期存活（重启 opencode 后还得在）、要能增删改查（换个账号、登出）、还涉及敏感安全。把这些统一收进一个有明确接口的库，比散落各处管理可靠得多。它把「一个凭据」抽象成两种形态的标签联合 Credential.Info：

OAuth
{type:"oauth", access, refresh, expires, methodID, metadata?}：会过期、能用 refresh 续的令牌
Key
{type:"key", key, metadata?}：一把直接的 API Key（不过期、不续）
Info = Union[OAuth, Key]
按 type 标签区分（toTaggedUnion("type")）——上层一视同仁
Stored
{id, integrationID, label, value:Info}：落库的一条凭据（属于哪个集成、贴个标签）
Interface
all/list/get/create/update/remove：对凭据的一整套 CRUD
CredentialTable
credential/sql：凭据落到 SQLite（同 L48 持久化）

注意这又是一次「异质塑同形」：OAuth 令牌（有 access/refresh/expires、要管过期）和 API Key（就一个字符串、永不过期）本是两种很不一样的东西，却被塑成同一个 Credential.Info 标签联合，进同一个有 create/get/remove 的库。上层要用凭据时，get(id) 拿到一个 Stored、看 value.type 是 oauth 还是 key，分别取用——差异被收进「一个 type 字段」，库本身对两种凭据一视同仁地存取。这和 L66 命令的「三源归一」、L36 工具的「统一模子」是同一种智慧：把异质塑成同形，复杂度收进塑形处，上层于是干净。Stored 还带了 integrationID 和 label——凭据不是孤零零的一串密钥，而是「属于某个集成（integration）、可以贴个人类可读标签」的一条记录，这让「我到底存了哪些账号的凭据」变得可列举、可管理（list(integrationID) 列某集成的全部凭据）。而它们最终落进 CredentialTable（SQLite，同 L48）——也就是说，你的登录态和密钥，和会话、消息一样，是被持久化管理的一等数据。

喂给 provider：凭据如何接到模型调用

凭据存下来不是目的，喂给 provider 去调模型才是。当一个 provider（如 OpenAI、Copilot）要发起请求时，它需要鉴权信息——凭据库里那条 Stored.value 就在此被取出：OAuth 的话用 access 令牌（作 Bearer，呼应 L35 Copilot provider 的 AuthOptions.bearer）、Key 的话直接用那个 key。这里你能再次看到「异质塑同形」的红利：provider 取凭据时，不必为「OAuth 还是 Key」写两套逻辑，只需 get(id) 拿到 Info、按 type 分一下叉即可——而「OAuth 要不要续、什么时候续」这种复杂，被关在凭据库自己那一侧消化掉了。这正好接上 L47「provider 插件」的接源能力（core/src/plugin/provider/openai-auth.ts 就是把这种鉴权接进 provider 的例子），也与 L46「MCP 的 OAuth」互为呼应——它们都是「把外部身份/凭据接进 opencode」的不同侧面。整条「登录→存库→取用」是这样：

设备码流登录
拿到令牌/Key
→
credential.create
存成 Stored（落 SQLite）
→
provider 要调模型
get(id) 取凭据
→
OAuth→Bearer access
/ Key→直接用

还有个体贴的工程细节：OAuth 令牌会过期，源码定了 eagerRefreshThreshold = Duration.minutes(5)——在令牌还差 5 分钟过期时就主动用 refresh_token 提前续，而不是等它过期、调用失败了再手忙脚乱地刷。这种「在出问题前就把它解决掉」的预判式刷新，让你在用 opencode 时几乎感觉不到「令牌过期」这件事的存在——它是那种「做对了你根本不会注意到、做错了你天天被打断」的体贴。把这一课嵌回全书，它处在 opencode「身份与凭据」这条横切线的几个连接点上：

L35 模型解析 / Copilot provider：provider 调模型要鉴权——OAuth 凭据的 access 作 Bearer 喂进去（AuthOptions.bearer），正是本课凭据的去处
L47 provider 插件：plugin/provider/openai-auth.ts 把这种鉴权接进 provider——「接源」能力（L47）与「凭据库」（本课）一里一外
L46 MCP 的 OAuth：MCP 服务器也可能要 OAuth——与本课「账号设备码登录」互为呼应，都是「把外部身份接进 opencode」的不同侧面
🗺️ 宏观图景
L68 讲 opencode 的登录与凭据。设备码流（account.ts）解决「CLI 没浏览器回调怎么 OAuth」：终端显示 user_code+URL（来自 DeviceAuth），你在任意有浏览器的设备授权，终端后台轮询令牌端点（DeviceTokenError.toPollResult 把 authorization_pending/slow_down/expired_token/access_denied 翻成轮询状态），成功拿 DeviceTokenSuccess{access, refresh, …}。「授权在别处、令牌落 CLI」被一个短码解耦。凭据库（credential.ts）把 OAuth（access/refresh/expires）与 Key 塑成同一个 Info 标签联合，进有 CRUD 的库（落 SQLite），eagerRefreshThreshold=5min 预判式续期。凭据再喂给 provider（L35/L47）调模型，与 L46 MCP OAuth 互补。又一次「异质塑同形」。这一课补上了全书一直缺的「身份与凭据」横切环：连模型、连服务前总得先登录、拿凭据，而 opencode 用「设备码 + 统一凭据库」干净地解决了它。
🔬 实现细节
opencode/src/account/account.ts：DeviceAuth{device_code, user_code, verification_uri_complete, expires_in, interval}；DeviceTokenSuccess{access_token, refresh_token, token_type:"Bearer", expires_in}；DeviceTokenError.toPollResult() 映射 authorization_pending→PollPending / slow_down→PollSlow / expired_token→PollExpired / access_denied→PollDenied / 其余→PollError；DeviceTokenRequest{grant_type, device_code, client_id} + TokenRefreshRequest{grant_type, refresh_token, client_id}；clientId="opencode-cli"，eagerRefreshThreshold=Duration.minutes(5)；DurationFromSeconds（秒↔Duration 编解码）；用 Cache/Clock 管 token 缓存与过期。core/src/credential.ts：OAuth{type:"oauth", access, refresh, expires, methodID, metadata?}、Key{type:"key", key, metadata?}、Info=Union([OAuth,Key]).toTaggedUnion("type")、Stored{id, integrationID, label, value:Info}、Interface{all,list,get,create,update,remove}、CredentialTable（credential/sql）。设计要点：设备码流用一组 Schema.Class 把 OAuth 设备授权的每个报文（DeviceAuth/请求/成功/错误）建模，toPollResult 把协议错误串翻成有名状态类，让轮询循环干净；凭据库用标签联合 + CRUD 接口把「两种凭据、长期存活、可增删」统一管理。两块合起来，就是 opencode「身份与凭据」这一横切关注点的完整落地。
🎯 本课要点
  • 设备码流（最点睛）：CLI 没浏览器回调，故不能用网页 OAuth。设备码把流程拆成「显示码 + 后台轮询」：终端显示 user_code+URL（DeviceAuth），你在任意有浏览器的设备授权，CLI 每隔 interval 轮询令牌端点；DeviceTokenError.toPollResult 把 authorization_pending/slow_down/expired_token/access_denied 翻成红绿灯，成功得 DeviceTokenSuccess。「授权在别处、令牌落 CLI」用短码解耦。
  • 凭据库（异质塑同形）：Credential.Info = Union[OAuth, Key] 按 type 标签区分——OAuth（access/refresh/expires、要续）与 Key（一个字符串、不过期）塑成同一个值，进有 all/list/get/create/update/remove 的库（Stored 落 CredentialTable/SQLite）。同 L66/L36「统一模子」。
  • 喂给 provider + 预判续期：取凭据喂 provider 调模型——OAuth 用 access 作 Bearer（呼应 L35 Copilot）、Key 直接用；接 L47 provider 插件、与 L46 MCP OAuth 互补。eagerRefreshThreshold=5min 在过期前主动用 refresh 续，让你感觉不到令牌过期。

🧪 自测 · 想一想为什么这么设计

1. 课里说设备码（device-code）流是为「CLI 没有浏览器回调」设计的。它的核心机制和解决的问题是？
  1. 把用户名密码明文存进配置文件，下次直接读
  2. 完全不需要授权，CLI 直接拿到令牌
  3. 把流程拆成「显示码 + 后台轮询」：终端显示 user_code+URL，你在任意有浏览器的设备授权，终端同时轮询令牌端点直到成功——「授权在别处、令牌落 CLI」被一个短码解耦，绕开了「CLI 收不了浏览器回调」这个死结
  4. 让 CLI 自己弹出一个浏览器窗口接收回调
看答案与解析 点击展开
答案：C。网页 OAuth 靠「浏览器回调」把授权结果送回应用，但 CLI 是个终端进程、没浏览器、没公网回调地址，这条路走不通。设备码（OAuth 2.0 Device Authorization Grant，本为智能电视等「输入受限」设备设计）把流程拆成两半：终端只显示 user_code+URL（来自 DeviceAuth），你在任意有浏览器的设备打开网址、输码、授权；同时终端揣着保密的 device_code 在后台轮询令牌端点，直到你授权完成就拿到令牌。「授权」在浏览器、「令牌」落 CLI，一个短码牵两头。所以不是「CLI 自己弹浏览器」「存明文密码」「不需授权」——这些都误解了设备码「把需浏览器的那步挪到别处」的本质。
2. 轮询令牌端点时，DeviceTokenError.toPollResult() 把 authorization_pending、slow_down 等 error 字符串翻成 PollPending、PollSlow 等状态类。这体现了什么？
  1. 加密 error 字符串，防止泄露
  2. 把所有错误都当成致命错误、直接终止轮询
  3. 把原始字符串升格成有名字的、有类型的值：OAuth 协议里「还没授权/轮太快/过期/被拒」都混在 error 字符串里，翻成状态类后轮询循环能干净地按状态分支，而不是到处比对魔法字符串（同 L20 RunError、L05「错误即值」）
  4. 把错误信息翻译成多国语言给用户看
看答案与解析 点击展开
答案：C。设备码轮询里，令牌端点对「还没好」用 HTTP 错误 + error 字符串回应：authorization_pending（继续等）、slow_down（放慢）、expired_token（过期重来）、access_denied（被拒结束）。这些不全是「真错误」——有的是「正常的还没好」。toPollResult 把这些字符串翻成有名状态类（PollPending/PollSlow/PollExpired/PollDenied/PollError），轮询循环就能 switch 这些状态、清清楚楚地决定「继续/放慢/重来/结束」，而不是满地 if (error === "...") 比对魔法字符串。这正是全书「把原始值升格成有类型的值」（L05 错误即值、L20 RunError）。所以不是翻译语言/全当致命/加密——核心是让轮询逻辑类型化、可读。
3. 凭据库（credential.ts）把 OAuth 令牌和 API Key 都塑成同一个 Credential.Info = Union[OAuth, Key]。这样做的好处，以及 eagerRefreshThreshold=5min 的作用是？
  1. eagerRefreshThreshold 是指令牌每 5 分钟就强制过期一次
  2. 凭据只存在内存里，重启 opencode 就没了
  3. 把 OAuth 和 Key 合并成一种，删掉 API Key 支持
  4. 异质塑同形：两种很不一样的凭据塑成同一个带 type 标签的值、进同一个 CRUD 库，上层取用时差异只剩「分一下 type 叉」（同 L66/L36）。eagerRefreshThreshold 让 OAuth 令牌在过期前 5 分钟主动用 refresh 续，避免「过期了调用才失败」

答案：D。OAuth 令牌（access/refresh/expires、要管过期）和 API Key（一个字符串、不过期）本是两种东西，却被塑成同一个 Credential.Info 标签联合（toTaggedUnion("type")），进同一个有 all/list/get/create/update/remove 的库——上层 get(id) 拿 Info、按 type 分一下叉即可，差异收进「一个 type 字段」（同 L66 三源归一、L36 统一模子）。eagerRefreshThreshold=Duration.minutes(5) 是预判式刷新：令牌还差 5 分钟过期就主动用 refresh_token 续，而非等过期、调用失败再补救，让你感觉不到过期。所以不是「删掉 Key 支持」「每 5 分钟强制过期」（恰相反，是提前续命），凭据也落 SQLite（CredentialTable）持久化、不是只在内存。

The first time you have opencode connect to an account/service in the terminal, you see this: the screen shows a short code (like WDJB-MJHT) and a URL, telling you to "open this URL in a browser and enter this code"; you do it, click authorize, return to the terminal, and it's already logged in. Behind this is the device-code OAuth flow—a login method designed for CLI/terminals that have no browser callback (opencode/src/account/account.ts). The obtained credential goes into a unified credential store (core/src/credential.ts), later fed to a provider to call the model. This lesson makes it clear: with no browser callback, how does a CLI complete OAuth? And the obtained keys/tokens—how are they stored and fed to providers? This is a piece cutting across the whole runtime yet never covered head-on in the prior 64 lessons—connecting to models and services in opencode always requires first "logging in, obtaining credentials," and this lesson fills in that "identity & credentials" link.

There are two insights worth taking away. First, the device-code flow decouples "where you authorize" from "where the token lands": traditional web OAuth relies on a "browser callback" to send the authorization result back to the app—but a CLI is a terminal process, with no browser and unable to receive a callback. The device-code trick: the terminal only shows a short code, you on any device with a browser (phone, another computer) open the URL, enter the code, authorize; meanwhile the terminal "polls" the token endpoint in the background until your side finishes, then it has the token. "Authorization" happens in the browser, "the token" lands in the CLI, the two ends linked by a short code. Second, credentials are unified into one type-tagged "value": whether an OAuth token (access+refresh+expires) or a direct API Key, both are shaped into the same Credential.Info tagged union, into one credential store with a CRUD interface—another "shape the heterogeneous into one form" (like L66/L36), so the upper layer "fetch a credential and feed the provider" needn't care which kind. Putting these two together, this lesson tells a complete story: opencode, as a tool that must connect to various external services (model providers, MCP, integrations), must solve "how does the user log in, how is the obtained key safely stored, then reliably used"—and its answer is a "device-code login + unified credential store" combination.

🎫 Analogy
The device-code flow is just like order-number pickup. You order at a busy restaurant (the CLI initiates login), the counter gives you a slip with an order number (user_code + that URL)—but you needn't stand dumbly waiting at the counter (the CLI can't "receive a callback"). You can take the number anywhere: enter it at a kiosk to confirm, get a manager's signature (on any device with a browser, open the URL, enter the code, authorize). Meanwhile the counter's number board keeps refreshing in the background: "not ready… not ready… (authorization_pending)" "too many people, refresh slower (slow_down)," until you confirm and the screen "dings" out your meal (access_token). The beauty: "confirmation" happens elsewhere, "pickup" completes at the counter, one number links both ends—neatly bypassing the deadlock that "the counter can't reach you on its own" (the CLI can't receive a callback). And the "meal" (credential), whether an OAuth monthly pass (expires, renewable) or a one-time key (API Key), is put into the same locker (credential store), retrievable anytime by a locker number (id); when the pass nears expiry, the shop even quietly renews it 5 minutes early (eager refresh), so you're never interrupted by "the pass expired."

Device-code flow: with no browser callback, how does a CLI do OAuth

Web apps do OAuth via a browser callback (redirect): the user clicks consent on the authorization page, the auth server redirects the result back to the app's pre-registered callback address. But a CLI/terminal is a process with no browser and no public callback address—this path is impossible. You can't have a program running in someone's terminal register a "jump back when authorized" URL (it has no URL, and no browser to catch the redirect). The device-code grant (OAuth 2.0 Device Authorization Grant) is the standard scheme for such "input/display-constrained" devices—originally for smart TVs, game consoles, and the like that are "hard to type passwords on, have no browser," and a CLI is exactly the same predicament. Its trick splits the flow into "show code + background polling," moving "the step that needs a browser" onto any convenient device at the user's hand:

Web OAuth (CLI can't use)

Redirect to the auth page → user consents → browser calls back with the code to the app. Relies on "the app can receive a browser callback"—the CLI has none.

Device-code flow (CLI uses this)

Terminal shows user_code+URL → user authorizes elsewhere → terminal polls the token endpoint until success. No callback needed, only "show + poll."

The source models every step. Step one exchanges for DeviceAuth: {device_code (secret, for polling), user_code (shown to a human to type), verification_uri_complete (auth URL), expires_in (when it expires), interval (how often to poll)}. Note the division of the two codes is deliberate: user_code is human-facing, short and easy to read (to hand-type on another device), device_code is machine-facing and secret (the CLI holds it to exchange for a token, never shown to a human). The terminal shows you user_code and the URL, holding device_code itself. Step two is polling: the terminal, every interval, takes device_code to ask the token endpoint (DeviceTokenRequest{grant_type, device_code, client_id}, client_id="opencode-cli"), returning either DeviceTokenSuccess{access_token, refresh_token, token_type:"Bearer", expires_in} or DeviceTokenError—and DeviceTokenError.toPollResult() translates OAuth's error strings into clear poll states, the whole polling loop's "traffic light." This toPollResult is a small design worth savoring: in OAuth, "not yet authorized," "polling too fast," "expired," "denied" all come back as error strings, jumbled together; toPollResult translates these strings into named state classes (PollPending/PollSlow/…), so the polling loop's code can cleanly switch on these states instead of comparing magic strings everywhere—another "promote a string/raw value into a typed value" (like L20 RunError, L05 errors-as-values):

error string→ poll statemeaning / what to do
authorization_pendingPollPendinguser hasn't finished authorizing—keep waiting, keep polling
slow_downPollSlowpolling too often—slow the interval
expired_tokenPollExpireddevice_code expired—start over
access_deniedPollDenieduser denied—end

Trace this "request → show → (you authorize elsewhere) → poll → get token" path and you see the device-code flow whole:

1 CLI requests, gets back DeviceAuth (device_code + user_code + URL + interval)
2 terminal shows: "open <URL>, enter user_code"
3 you on a phone/browser open the URL, enter the code, click authorize (this step is outside the CLI)
4 meanwhile the CLI polls every interval: Pending…Pending…(after you authorize)→ DeviceTokenSuccess
5 got access_token+refresh_token, login complete; auto-renews 5 minutes before expiry (eagerRefreshThreshold)

The credential store: OAuth tokens and API keys, one locker

What login obtains must be stored and reused repeatedly—that's core/src/credential.ts's job. Why a dedicated "credential store" rather than tossing the token into some variable? Because credentials have their own lifecycle and diversity: multiple forms (OAuth token / API Key), long-lived (still there after restarting opencode), need CRUD (switch accounts, log out), and involve security sensitivity. Gathering these into one store with a clear interface is far more reliable than managing them scattered. It abstracts "a credential" into a two-form tagged union Credential.Info:

OAuth
{type:"oauth", access, refresh, expires, methodID, metadata?}: an expiring token renewable via refresh
Key
{type:"key", key, metadata?}: a direct API Key (no expiry, no renewal)
Info = Union[OAuth, Key]
distinguished by the type tag (toTaggedUnion("type"))—the upper layer treats them uniformly
Stored
{id, integrationID, label, value:Info}: a stored credential (which integration, with a label)
Interface
all/list/get/create/update/remove: a full CRUD over credentials
CredentialTable
credential/sql: credentials land in SQLite (like L48 persistence)

Note this is again "shape the heterogeneous into one form": an OAuth token (with access/refresh/expires, managing expiry) and an API Key (just a string, never expires) are two very different things, yet shaped into the same Credential.Info tagged union, into one store with create/get/remove. When the upper layer wants a credential, get(id) fetches a Stored, checks whether value.type is oauth or key, and uses each accordingly—the difference tucked into "one type field," the store itself storing/fetching both uniformly. This is the same wisdom as L66 commands' "three sources, one form" and L36 tools' "one mold": shape the heterogeneous into one form, complexity tucked into the shaping spot, the upper layer clean. Stored also carries integrationID and label—a credential isn't a lone key string but a record "belonging to some integration, with a human-readable label," making "which accounts' credentials have I stored" enumerable and manageable (list(integrationID) lists one integration's credentials). And they ultimately land in CredentialTable (SQLite, like L48)—meaning your login state and keys, like sessions and messages, are first-class, persistently-managed data.

Feeding the provider: how credentials reach a model call

Storing credentials isn't the goal—feeding the provider to call the model is. When a provider (e.g. OpenAI, Copilot) makes a request, it needs auth info—that Stored.value in the store is fetched here: for OAuth, use the access token (as Bearer, echoing L35 Copilot provider's AuthOptions.bearer); for Key, use the key directly. Here you again see the dividend of "shape the heterogeneous into one form": when a provider fetches a credential, it needn't write two code paths for "OAuth vs Key," just get(id) for the Info and branch once on type—while the complexity of "whether OAuth needs renewing, when" is digested on the credential store's own side. This connects to L47 "provider plugins'" source-attaching ability (core/src/plugin/provider/openai-auth.ts is exactly an example of plugging such auth into a provider), and echoes L46 "MCP's OAuth"—both are facets of "plug external identity/credentials into opencode." The whole "login → store → use" goes like:

One more thoughtful engineering detail: OAuth tokens expire, and the source sets eagerRefreshThreshold = Duration.minutes(5)—when the token is 5 minutes from expiry, it proactively renews early with the refresh_token, rather than waiting for expiry, a failed call, then scrambling to refresh. This "predictive refresh" of "solve it before it becomes a problem" makes you barely notice "token expiry" while using opencode—the kind of thoughtfulness "you never notice when done right, but get interrupted by daily when done wrong."
