其余生态一览Ecosystem tour

前面 68 课，我们几乎全程待在 opencode 的引擎舱里——Effect 地基、Session 主循环、Context Epoch、LLM 协议、工具、持久化、扩展。但这台引擎的外围，还围着一整圈我们一直没正眼看过的东西：你能用的不止终端 TUI，还有网页版、Electron 桌面版；你的会话能分享成一个链接让别人在线只读旁观；还有 Slack 机器人、企业版、云基础设施……这一课不深挖任何一个的实现，而是做一次鸟瞰：把这一整圈生态摆开，看清它们各自是什么、以及它们如何全都连着同一个内核。作为全书最后一部分的收官一课，它的任务不是再塞给你一个机制，而是把你前面学的所有引擎零件，放回它们所驱动的那一整辆「车」里看一眼——让你对 opencode 的全貌，有一个完整、立体的收尾印象。读完你会发现，这一圈看似五花八门的东西，其实都站在前面几课打下的同一块地基上。

这一课最值得带走的洞见只有一个，但它把全书好几条线收束到了一起：opencode 是「一台引擎、多副面孔与接口」。还记得第 9 课「server 拥有一切」、第 56 课「数据与表现分离」、第 61 课「ACP 让编辑器驱动它」吗？

这一课就是这三条线的最终展开：内核（那个常驻、自描述、广开门户的 server）只有一个，而围着它，长出了 TUI、网页、桌面、Slack、可分享的云视图……这么多张面孔与接入点。它们之所以能共存且不重复造轮子，全靠它们共享同一套内核、同一份 SDK、同一条事件流（L11/L12）。理解这点，你就理解了 opencode 为什么能「一处内核、处处可达」——而你前面学的每一块引擎机制，正是这一整圈生态共同的底座。

换句话说，这一课不会教你任何「新机制」，它教你的是一种看待全局的眼光：当你面对一个长出了一大圈周边的系统时，别被表面的五花八门迷惑，去找那个大家共享的内核——找到它，整个生态的逻辑就一目了然了。

🚗 生活类比
把 opencode 的内核 server 想象成一台发动机（第 9 课说它「拥有一切」）。围着这同一台发动机，厂家造出了不同的车身：终端 TUI 是那台皮实的越野车、网页版是家用轿车、Electron 桌面版是带豪华内饰的同款轿车——它们外形手感各异，但动力总成（SDK + 事件流）是同一套，踩油门的反馈都来自那同一台发动机。Slack 机器人像给发动机接了个远程遥控器，你在聊天里发个指令它就替你开。会话分享 + 云同步，像给车装了个实时位置直播：你在本地开车（单写入，第 65 课），别人扫个链接就能在线只读旁观你开到哪了。而企业版、云基础设施，是把这套东西批量部署、接进公司流程的「车队管理」。 关键认知始终是那一句：无论多少种车身、多少个遥控器，底下都是同一台发动机。正因如此，厂家想多出一款车、多接一个遥控器，都不必重造发动机——这就是「一台引擎养活一整个生态」的省力之处，也是你前面 64 课为什么值得读：你学的全是那台发动机的内部构造，而这一课，是退后一步，看看这台发动机到底驱动了多少东西。

多前端：一个 server，多副面孔

opencode 最根本的架构决定（第 3、9 课）是：内核跑成一个 server，所有「界面」都只是连上它的客户端。这一决定的红利，在这一圈前端上体现得淋漓尽致——想象一下若没有这个决定会怎样：每做一种前端，就得把「会话怎么管、工具怎么跑、上下文怎么拼」重写一遍，五种前端就是五套貌合神离、各自有 bug 的内核。而 opencode 选择「一个 server、多客户端」，于是：

没有「一个 server」（设想）

每种前端各自实现一套会话/工具/上下文逻辑——五副面孔=五套内核，重复、漂移、各有各的 bug，加一种前端=重写一遍核心。

opencode：一个 server、多客户端

内核只有一个，前端只是连上它的客户端。加一种面孔，只需写「怎么把数据画出来」，内核一行不用动。

同一个 server，被好几种完全不同的前端共享：

表现层（多副面孔）　TUI（终端）· web app · Electron 桌面 · Slack · 可分享云视图
连接层（同一套）　生成的 SDK（L12）+ SSE 事件流（L11）——所有前端共用
内核（只有一个）　server：Session、工具、Context、LLM…（L09「拥有一切」）

具体看这几副「面孔」：web app（packages/app，@opencode-ai/app）是用 SolidJS 写的网页 UI，它直接依赖 @opencode-ai/sdk 和 @opencode-ai/core——和 TUI 一样，靠 SDK + 事件流连 server，只是把界面画在浏览器里而非终端。Electron 桌面版（packages/desktop，@opencode-ai/desktop）干脆把这个 web app 用 Electron 壳子包成原生桌面应用（依赖 electron-updater 等，带自动更新）——所以桌面版几乎不是「另写一个 app」，而是「给 web app 套个壳」。这正是第 56 课「数据与表现分离」的终极兑现：一份内核数据，能渲染成终端、网页、桌面好几副皮，每多一副都不必重写内核。这种「壳子套壳子」的复用还体现在桌面版上——它甚至不是「web 之外再写一遍」，而是直接把 web app 装进 Electron，连界面代码都省了重写。各前端各管一摊、却同源：

packages/app
SolidJS web UI，依赖 sdk+core（网页那副面孔）
packages/desktop
Electron 包住 web app，带自动更新（桌面那副面孔）
cli/tui
终端 TUI（L52–56），同样连 server
packages/slack
@slack/bolt + sdk：Slack 里驱动 opencode
packages/enterprise
SolidStart 企业版 web（接公司流程）
packages/ui
共享 UI 组件（shiki/katex/diff…，多前端复用）

分享与云同步：把会话投影到云端

第 65 课讲过 opencode 的「单写入 + 事件溯源」同步底座：本地一台设备是写入者，事件带单调 seq 全序排好。会话分享（opencode/src/share）就建在它之上：一个 SessionShare.Service 提供 share(sessionID) → { url } 和 unshare——把一个本地会话变成一个可分享的链接。注意这件事之所以能如此轻巧，正是因为第 65 课那块地基已经备好：会话本就是一条带全序号、可重放的事件流，「分享」无非是把这条流转发到云端一个中转站，让没有你本地进程的人也能订阅、重放。

链接背后是 packages/function 里的一台云端 Cloudflare Durable Object（SyncServer extends DurableObject<Env>）：它用 WebSocket + 自带存储，接收你本地推上去的会话事件（按 session/ 前缀存），再广播给所有打开了那个链接的旁观者。

为什么用 Durable Object？因为它天然适合「一个会话对应一个有状态、长连接的中转点」——每个分享的会话就是一个 Durable Object 实例，存着该会话的事件、维护着旁观者们的 WebSocket。于是别人点开链接，就能在线只读地看你的会话实时进展——这正是第 65 课「单写入者写、其他设备重放追平」在云端的落地：你的浏览器/手机是「其他设备」，云 Durable Object 是中转的事件日志。整条「本地会话 → 分享 → 云端 → 他人只读」是这样：

（你是单写入者 L65）

接入与运维：触发器、Slack、企业与基础设施

除了「人去用」的前端，opencode 还有几块「接进流程/规模化」的生态。Integration（集成）（core/src/integration.ts）是一套声明式的连接与触发机制（IntegrationSchema、IntegrationConnection、MethodID）——它让外部系统/事件能按声明的条件接进会话，是把 opencode 嵌进自动化流程的接口（也呼应第 68 课的凭据：连外部服务要存凭据）。你可以把它理解成「当某个外部条件满足时，自动触发一段 opencode 的活」的声明式接线板：连接定义「连的是谁、用什么凭据」，触发定义「什么时候动」。Slack 机器人（packages/slack）用 @slack/bolt + SDK，把 opencode 接进团队的 Slack——你在频道里发指令，它就替你跑，这是「多副面孔」里很特别的一副：界面不是你自己画的 UI，而是借了 Slack 这个现成的聊天界面。企业版（packages/enterprise，SolidStart）面向公司场景，把这套能力按企业需求重新包装。基础设施（infra/，SST）则用「基础设施即代码」描述云端部署——让「这一切怎么跑在云上」本身也成为可版本化、可复现的代码。这些块各异，但共享同一个底层立场——都连着那一个 server：

Integration（core/src/integration.ts）：声明式触发器/连接，把外部事件接进会话——opencode 嵌进自动化流程的接口
Slack / 企业版（packages/slack / enterprise）：把 opencode 接进团队聊天与公司流程，仍经 SDK 连 server
基础设施（infra/，SST）：基础设施即代码，描述这一切如何部署上云

最后用一张表把这一整圈生态收拢——你不必记住每个包的细节，只需记住它们都围着同一个内核、靠同一套 SDK/事件连接这件事：

生态块是什么与内核的关系
app / desktopweb UI / Electron 桌面（套壳 web）经 SDK+事件连 server（另几副面孔）
share + function会话分享 + 云 Durable Object 同步把本地单写入 sync（L65）投影到云端只读
integration声明式连接/触发器把外部事件接进会话
slack / enterpriseSlack bot / 企业版 web把 opencode 接进团队与公司流程
infraSST 基础设施即代码描述云端部署

至此，opencode 的全貌真正完整了：一个常驻、自描述、广开门户的内核 server（L09），用一套生成的 SDK 与事件流（L11/L12）向外敞开；围着它，长出了 TUI、网页、桌面这些面孔（L52–56、本课），ACP/插件/LSP 这些扩展（L57–61），以及分享、集成、Slack、企业、基础设施这一整圈接入与运维。而这一切之所以成立，根都在前面那些引擎课——正因内核被做成了「数据与表现分离、广开门户」的样子，它才能长出这么多副面孔而不重复造轮子。一台引擎，养活一整个生态。这也正是读源码最值得带走的一种眼光：面对任何一个庞大的系统，先别数它有多少零件，先找到那个大家共享的核心，再看每一圈周边如何围着它生长——抓住了核心，再繁茂的生态也有了脉络。这趟从「opencode 是什么」到「它的整个生态如何围着一个内核生长」的旅程，到这里就走完了。

🎯 本课要点
  • 一台引擎、多副面孔：内核 server 只有一个（L09），TUI/web(app)/桌面(desktop，Electron 套壳 web、带自动更新) 都经同一套 SDK（L12）+ 事件流（L11）连它——第 56 课「数据与表现分离」的终极兑现：一份内核，多副皮，每多一副不必重写内核。

2. 会话分享（share(sessionID)→{url}）让别人点开链接就能在线只读看你的会话。它为什么能做得这么轻巧？
  1. 因为分享时会让旁观者也能写你的会话，多人共写

  3. 因为它把你的整个本地数据库实时上传到公网供人下载
  4. 因为它给每个旁观者都跑了一个完整的 opencode 内核
看答案与解析 点击展开
答案：B。分享之所以轻巧，根在 L65 那块地基：会话已经是一条单写入、带单调 seq 全序、可重放的事件流。opencode/src/share 的 share(sessionID)→{url} 把这条流推到 packages/function 的一台 Cloudflare Durable Object（SyncServer extends DurableObject，WebSocket+存储，按 session/ 前缀存）中转，旁观者打开链接就经 WebSocket 订阅、重放事件追平——你的浏览器/手机就是 L65 说的「其他（只读）设备」。注意是只读（单写入约束没变），不是多人共写；也不需要给每个旁观者跑完整内核（他们只重放事件）。这正是「好的底层抽象会在意想不到的上层开花」（同 L61 move-session 建在 Location 解耦上）。

Over the previous 68 lessons we stayed almost entirely in opencode's engine bay—the Effect foundation, the Session main loop, Context Epoch, the LLM protocol, tools, persistence, extensions. But around this engine's periphery is a whole ring of things we never looked at squarely: what you can use isn't just the terminal TUI, but a web version, an Electron desktop version; your session can be shared as a link for others to watch read-only online; there's also a Slack bot, an enterprise edition, cloud infrastructure… This lesson doesn't dig into any one's implementation but takes a bird's-eye view: lay out this whole ecosystem ring and see clearly what each is and how they all connect to the same core. As the closing lesson of the book's final part, its task isn't to hand you another mechanism but to put all the engine parts you've learned back into the whole "car" they drive and take a look—giving you a complete, three-dimensional closing impression of opencode. By the end you'll find this seemingly miscellaneous ring all stands on the same foundation laid by the earlier lessons.

There's only one insight worth taking away, but it gathers several of the book's threads together: opencode is "one engine, many faces and interfaces." Remember Lesson 9's "the server owns everything," Lesson 56's "data separated from presentation," Lesson 61's "ACP lets editors drive it"? This lesson is those three threads' final unfolding: the core (that resident, self-describing, doors-wide-open server) is one, and around it grow TUI, web, desktop, Slack, a shareable cloud view… so many faces and entry points. They coexist without reinventing wheels entirely because they share the same core, the same SDK, the same event stream (L11/L12). Grasp this and you grasp why opencode is "one core, reachable everywhere"—and every engine mechanism you learned earlier is this whole ecosystem's shared foundation. In other words, this lesson teaches no "new mechanism"; it teaches a way of seeing the whole: facing a system that has grown a large surrounding ring, don't be dazzled by the surface variety—find the core everyone shares—find it and the whole ecosystem's logic becomes clear at a glance.

🚗 Analogy
Picture opencode's core server as an engine (Lesson 9 says it "owns everything"). Around this same engine, the maker built different car bodies: the terminal TUI is the rugged off-roader, the web version a family sedan, the Electron desktop a luxe-interior version of that same sedan—different in look and feel, but the powertrain (SDK + event stream) is the same, the throttle feedback all from that same engine. The Slack bot is like a remote control wired to the engine—you send a command in chat and it drives for you. Session sharing + cloud sync is like fitting the car with a live location broadcast: you drive locally (single writer, Lesson 65), and others scan a link to watch read-only online where you've driven. The enterprise edition, cloud infrastructure are the "fleet management" of deploying this in bulk and wiring it into company processes. The key realization is always that one line: however many bodies, however many remotes, underneath is the same engine. Precisely because of this, the maker can add a model or wire up another remote without rebuilding the engine—that's the leverage of "one engine sustains a whole ecosystem," and why the earlier 64 lessons were worth reading: you learned that engine's internals, and this lesson steps back to see how much that engine actually drives.

Multiple frontends: one server, many faces

opencode's most fundamental architectural decision (Lessons 3, 9) is: the core runs as a server, all "interfaces" are merely clients connecting to it. That decision's dividend shows vividly across this ring of frontends—imagine if there were no such decision: every frontend would reimplement "how sessions are managed, how tools run, how context is assembled," five frontends being five superficially-alike, individually-buggy cores. opencode chose "one server, many clients," so:

Without "one server" (hypothetical)

Each frontend implements its own session/tool/context logic—five faces = five cores, duplicated, drifting, each buggy; adding a frontend = rewriting the core.

opencode: one server, many clients

The core is one, frontends are merely clients connecting to it. Adding a face only needs "how to draw the data," the core untouched.

One server, shared by several completely different frontends:

Presentation (many faces)　TUI (terminal) · web app · Electron desktop · Slack · shareable cloud view

Look at these "faces" concretely: the web app (packages/app, @opencode-ai/app) is a web UI written in SolidJS, depending directly on @opencode-ai/sdk and @opencode-ai/core—like the TUI, it connects to the server via SDK + event stream, just drawing the UI in a browser rather than the terminal. The Electron desktop (packages/desktop, @opencode-ai/desktop) simply wraps this web app into a native desktop app with Electron (depending on electron-updater etc., with auto-update)—so the desktop version is almost not "writing another app" but "putting a shell on the web app." This is Lesson 56's "data separated from presentation" ultimately cashed out: one set of core data renders into several skins—terminal, web, desktop—each new one needing no core rewrite. This "shell within a shell" reuse shows on the desktop too—it's not even "written again beyond web" but directly packs the web app into Electron, sparing even rewriting the UI code. Sharing & cloud sync: projecting a session to the cloud

Lesson 65 covered opencode's "single-writer + event-sourcing" sync foundation: one local device is the writer, events totally ordered by a monotonic seq. Session sharing (opencode/src/share) is built atop it: a SessionShare.Service provides share(sessionID) → { url } and unshare—turning a local session into a shareable link. Note this is so lightweight precisely because Lesson 65's foundation is already in place: a session is already a totally-numbered, replayable event stream, so "sharing" is no more than forwarding that stream to a cloud relay for people without your local process to subscribe and replay.

Behind the link is a cloud Cloudflare Durable Object in packages/function (SyncServer extends DurableObject<Env>): it uses WebSocket + its own storage to receive the session events you push up (stored by session/ prefix), then broadcasts to everyone who opened that link. Why a Durable Object? Because it naturally fits "one session ↔ one stateful, long-connection relay"—each shared session is a Durable Object instance, storing that session's events and maintaining the watchers' WebSockets.

So others click the link and can watch your session's live progress, read-only, online—exactly Lesson 65's "the single writer writes, other devices replay to catch up" landed in the cloud: your browser/phone is an "other device," the cloud Durable Object the relaying event log. The whole "local session → share → cloud → others read-only" goes like:

Integration & ops: triggers, Slack, enterprise & infrastructure

Beyond the "people use it" frontends, opencode has a few "wire into processes / scale up" ecosystem pieces. Integration (core/src/integration.ts) is a declarative connection & trigger mechanism (IntegrationSchema, IntegrationConnection, MethodID)—it lets external systems/events plug into a session per declared conditions, the interface for embedding opencode into automation flows (also echoing Lesson 68's credentials: connecting external services needs stored credentials). Think of it as a declarative patch panel of "when some external condition is met, auto-trigger a piece of opencode work": the connection defines "who's connected, with what credentials," the trigger defines "when it fires."

The Slack bot (packages/slack) uses @slack/bolt + SDK to wire opencode into the team's Slack—you send a command in a channel and it runs for you, a special "face" among the many: the interface isn't a UI you drew but borrows Slack's ready-made chat interface. The enterprise edition (packages/enterprise, SolidStart) targets company scenarios, repackaging this capability per enterprise needs.

The infrastructure (infra/, SST) describes cloud deployment with "infrastructure as code"—making "how all this runs in the cloud" itself versionable, reproducible code. These pieces differ, yet share one underlying stance—all connect to that one server:

Integration (core/src/integration.ts): declarative triggers/connections, plugging external events into a session—opencode's interface into automation flows
Slack / enterprise (packages/slack / enterprise): wire opencode into team chat and company processes, still connecting to the server via the SDK
Infrastructure (infra/, SST): infrastructure as code, describing how all this deploys to the cloud

Finally, gather this whole ecosystem ring in one table—you needn't memorize each package's details, only the fact that they all surround the same core, connecting via the same SDK/events:

With that, opencode's whole picture is truly complete: a resident, self-describing, doors-wide-open core server (L09), opening outward via a generated SDK and event stream (L11/L12); around it grow the faces—TUI, web, desktop (L52–56, this lesson)—the extensions—ACP/plugins/LSP (L57–61)—and this whole ring of integration & ops: sharing, integration, Slack, enterprise, infrastructure. And all of it holds precisely because of those earlier engine lessons—because the core was built "data separated from presentation, doors wide open," it can grow so many faces without reinventing wheels. One engine, sustaining a whole ecosystem. This is also the most worthwhile lens to take from reading source: facing any large system, don't count its parts first, find that shared core, then see how each surrounding ring grows around it—grasp the core and even the most luxuriant ecosystem gains a thread.
